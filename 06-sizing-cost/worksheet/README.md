# 계산 워크북 (sizing-workbook.xlsx)

> 기반 버전은 [상위 README의 기반 버전 기준](../README.md#기반-버전-source-of-truth)을 참조하세요.

가이드(01~08, 부록 A1~A3)의 산식을 **라이브 수식 스프레드시트**로 옮긴 계산기입니다. 입력값만 바꾸면 GPU·노드·클러스터·스토리지·TCO 골격이 자동 재계산됩니다.

## 파일

| 파일 | 내용 |
|---|---|
| [`sizing-workbook.xlsx`](sizing-workbook.xlsx) | 계산 워크북(표지·입력·산정·TCO 4시트) |
| [`build_workbook.py`](build_workbook.py) | 워크북 생성 스크립트(재현·수정용, 모든 산식의 원본) |

## 빠른 시작 — 무엇을 채우나

입력 항목은 약 40개지만, **현업이 직접 정해야 하는 자사 고유값은 10개 안팎**입니다. 나머지는 모델 사양 조회값이거나 플랫폼 표준 기본값이라 대개 그대로 둡니다. 아래 **A 그룹만 채워도 1차 윤곽**이 나옵니다.

## 입력값 4분류 — 무엇을 꼭 채우고 무엇은 그대로 두나

### A. 반드시 자사 값으로 (핵심 — 보통 10개 안팎)

현업이 알고 있거나 결정해야 하는 고유값입니다.

| 입력 | 어디서 얻나 |
|---|---|
| 대상 사용자 수 | 서비스 대상 인원 |
| DAU 비율 | 사내 도구 사용 패턴(사내 도구는 보통 0.2~0.4) |
| 1인당 일 질의 | 업무 분석·추정 |
| 파라미터 수 | 선택한 모델 규모(예: 8B·70B) |
| GPU 메모리 | 보유·도입 예정 GPU 사양 |
| 문서 코퍼스 원문(GB) | 대상 문서량 |
| 벡터 수 | 코퍼스 청크 수 추정 |
| 호스트당 코어·소켓 수 | 서버 스펙시트 |
| 라이선스 대상 호스트 수 | 도메인 구성 |

### B. 모델 사양 — 조회값 (측정이 아니라 모델 파일에서 찾음)

| 입력 | 어디서 |
|---|---|
| 레이어 수·KV 헤드 수·head_dim | 모델 `config.json` (부록 A2.5) |
| 정밀도 바이트 | FP16=2, FP8 KV=1 등 양자화 선택에 따라 |
| 컨텍스트·평균 출력 토큰 | 서비스 설계값(RAG는 보통 4K 안팎) |

### C. 표준 기본값 — 보통 그대로 둠 (플랫폼 상수·HA 규칙)

업무시간 8h, 피크/평균비 3, 버스트 0.3, gpu_memory_utilization 0.9, PAIF 최소 GPU 호스트 3, 컨트롤 플레인 3, 일반 워커 3, 노드 헤드룸 0.2, vSAN 보호배수 1.5, HNSW 배수 2, 전처리 배수 1.5, 모델 보존 버전 3, Supervisor 클러스터·노드 한도 등. 근거는 부록 A2. 특별한 사유가 없으면 바꾸지 않습니다.

### D. 견적·실측으로 갈음 (워크북의 노란 칸 — 1차 가정치)

| 입력 | 왜 노란 칸인가 |
|---|---|
| 1차 처리량 가정(tok/s) | 모델·하드웨어마다 달라 예산 윤곽 전용. 확정은 PoC·부하시험 실측으로 갈음([06.5](../docs/06-capacity-planning.md)) |
| NIM 컨테이너 이미지(GB) | 모델별로 상이 — 확인 필요 |
| 소켓당 코어 최소수량 | 라이선스 정책 — [부록 A3](../appendix/A3-rfq-quote-checklist.md) 견적 |

> 측정이 어려운 값(특히 처리량)은 1차 가정치로 **구매 전 윤곽**만 잡고, 확정 사이징은 PoC 실측으로 갈음합니다. 이 워크북은 "얼마나 사야 하나"를 가늠하는 도구이지 SLA를 보증하는 도구가 아닙니다.

## 계산 흐름 — 입력이 어떻게 결과가 되나

'산정' 시트의 셀 수식을 풀어 쓴 것입니다(xlsx를 열지 않아도 무엇이 어떻게 계산되는지 보이도록). 모든 산식의 원본은 [`build_workbook.py`](build_workbook.py)이며, 근거 가이드 절을 함께 표기했습니다.

```
[워크로드 입력] ─▶ 동시성  (가이드 A2.1)
   DAU         = 사용자수 × DAU비율
   일요청       = DAU × 1인당 일질의
   피크 QPS     = 일요청 / (업무시간×3600) × 피크비
   설계 동시성   = ROUNDUP( 피크QPS × 요청처리시간 × (1+버스트) )

[모델·컨텍스트 입력] ─▶ GPU 메모리  (가이드 02)
   가중치        = 파라미터수 × 정밀도바이트
   토큰당 KV     = 2 × 레이어 × KV헤드 × head_dim × KV정밀도 / 1GB
   총 KV        = 토큰당KV × 컨텍스트 × 설계동시성
   소요 VRAM     = 가중치 + 총KV + 오버헤드
   가용 VRAM/Replica = Replica당GPU × GPU메모리 × util   ─▶ 적재 가능? 판정

[처리량·메모리] ─▶ Replica ─▶ GPU  (가이드 02.5 / A1)
   Replica(처리량) = ROUNDUP( 피크QPS × 출력토큰 / 처리량가정 )
   Replica(메모리) = ROUNDUP( 설계동시성 / Replica당 수용동시성 )
   총 Replica     = MAX(처리량, 메모리) + N+1
   총 GPU         = 총Replica × Replica당GPU + 임베딩GPU

[GPU] ─▶ 노드 ─▶ 클러스터  (가이드 04)
   GPU노드      = MAX( ROUNDUP(총GPU / 노드당GPU), PAIF 최소 3 )
   클러스터 노드  = GPU노드 + 일반워커 + 컨트롤플레인
   총 노드       = 클러스터노드 × 클러스터수      (Supervisor 한도 점검)

[스토리지 입력] ─▶ 가용 용량  (가이드 05)
   논리합        = 모델(가중치×보존버전) + NIM이미지 + 코퍼스×전처리배수
                 + 벡터(원시 + HNSW배수) + 로그
   필요 가용      = 논리합 × 보호배수 / 절감배수

[라이선스·총GPU] ─▶ TCO  (가이드 07)
   라이선스 코어  = MAX(호스트코어, 소켓×최소수량) × 호스트수
   NVAIE GPU    = 총GPU
   비용         = 각 수량 × 견적 단가(노란 칸, A3)
```

핵심은 **설계 동시성**과 **소요 VRAM** 두 값입니다. 동시성이 필요한 GPU 수를 끌어올리고, VRAM이 모델 적재 가능 여부를 가릅니다. 나머지 결과는 대부분 이 둘에서 파생됩니다. 입력을 바꿨을 때 결과가 이상하면 이 두 값을 먼저 확인하세요.

## 사용법

1. `sizing-workbook.xlsx`를 Excel 또는 LibreOffice Calc로 엽니다(열면 수식이 자동 재계산됩니다).
2. **입력** 시트의 **파란 셀**을 자사 값으로 바꿉니다(위 A 그룹부터).
3. **산정**·**TCO** 시트에서 결과를 확인합니다.

## 색상 약속

- 파란 글씨: 입력(여기를 바꿔 시나리오 조정)
- 검은 글씨: 수식·계산(직접 수정 금지)
- 노란 배경: 확인 필요 / 1차 가정치 / 단가 견적 입력 칸(위 D 그룹)

## 기본 예시

기본 입력은 [docs/08 레퍼런스 시나리오](../docs/08-reference-scenario.md)(사내 5,000명 한국어 RAG 챗봇)와 동일하며, 산출 결과도 동일합니다: **GPU 약 3장 / 클러스터 9 노드 / 가용 스토리지 약 0.6TB / 라이선스 코어 192**.

## 중요(반드시 읽기)

- **처리량·동시성 등 핵심 입력의 기본값은 예산 추정 전용 1차 가정치**(부록 A1)입니다. 확정 사이징은 PoC·부하시험 실측값으로 갈음하세요([06.5](../docs/06-capacity-planning.md#65-poc--파일럿--프로덕션-용량-로드맵)).
- **단가는 제시하지 않습니다.** TCO 시트의 노란 칸은 [부록 A3 견적 요청 체크리스트](../appendix/A3-rfq-quote-checklist.md)로 채우고 출처를 기록하세요.
- 모든 수치는 어림이며 환경·릴리스별로 달라집니다. 적용 전 공식 문서·견적·실측으로 재확인하세요.

## 적용 범위와 한계

- **단일 모델을 (Replica당 GPU)×GPU 메모리에 적재하는 추론 워크로드**용 1차 도구입니다. 모델이 단일 Replica에 안 들어가면 '산정' 시트가 "**적재불가**"로 표시되며, 입력의 "**Replica당 GPU**"(텐서 병렬 GPU 수)를 늘리면 됩니다(예: 70B FP16은 4 부근).
- "가용 VRAM/Replica = Replica당 GPU × GPU 메모리 × util"로 멀티-GPU(텐서 병렬)를 반영합니다. "소요 VRAM" 행은 1 Replica가 전체 동시성을 받는 경우의 참고치이며, Replica가 여러 개로 나뉘면 상한 정보로만 보세요.
- 다중 모델 동시 서빙·학습/파인튜닝·MIG 세부 분할·멀티테넌트 합산은 범위 밖입니다.

## 재생성

```bash
pip install openpyxl
python build_workbook.py   # sizing-workbook.xlsx 재생성
```

> 저장소의 `sizing-workbook.xlsx`는 **열자마자 값이 보이도록 계산값(캐시)을 포함**해 둔 버전입니다(기본 시나리오 = 5,000명 RAG). `build_workbook.py`로 재생성하면 수식만 있는(캐시 없는) 파일이 되며, Excel/LibreOffice로 열면 자동 재계산됩니다. 입력을 바꾸면 저장 시 계산값도 함께 갱신됩니다.
