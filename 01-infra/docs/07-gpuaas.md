# 07 — GPUaaS (PAIF GPU 자원 서비스)

> 기반 버전은 [README 버전 기준 문서](../README.md#기반-버전-source-of-truth)를 참조하세요.
> 이 문서는 PAIF를 **AI 애플리케이션(PAIS) 관점이 아니라, GPU 자원 자체를 사내·계열사에 서비스로 제공하는 GPUaaS 관점**에서 다룹니다. 문서 01–06이 "AI를 어떻게 올리나"였다면, 이 문서는 "GPU를 어떻게 빌려주나"입니다.

> **고지:** 본 문서의 시나리오는 가상의 일반 조직·그룹을 가정하며 **특정 기업·고객사를 지칭하지 않습니다.** 과금·라이선스·분할 관련 수치와 기능은 작성 시점(2026-06) 기준이며, 적용 전 공식 문서로 재확인하시기 바랍니다.

---

## 7.1 GPUaaS란 / PAIF에서의 위치

대부분의 조직에서 GPU는 희소하고 비쌉니다. 팀마다 GPU 서버를 따로 사면 활용률은 낮고 관리는 분산됩니다. **GPUaaS는 GPU를 공용 풀로 모아, 사내 팀·계열사가 셀프서비스로 빌려 쓰게 하는 운영 모델**입니다. 클라우드의 GPU 인스턴스를 사내(또는 그룹) 안에서 재현하는 것입니다.

**PAIF 코어 기능 계층**이 이 GPUaaS의 기반입니다. VCF와 PAIF가 GPU 자원 풀링·분할·격리·셀프서비스(VCF Automation)·과금·관측(VCF Operations)을 제공하고, 그 위에서 테넌트가 자신의 AI 워크로드를 올립니다.

**GPUaaS·PAIS와 PAIF의 관계(한 줄 정리):** GPUaaS는 별도의 아키텍처 계층이 아니라, **PAIF 코어 기능 계층의 GPU 자원**(GPU enablement·VM Class·카탈로그·쇼백/차지백)을 사내·계열사에 셀프서비스로 빌려주는 **운영 모델**입니다(공식 컴포넌트가 아닌 가이드 개념, [문서 02 §2.1](02-architecture.md#21-전체-계층-구조)). 그 위에서 모델·서빙을 테넌트에게 맡기면 전통적 GPUaaS(Tier 1)이고, 플랫폼이 **PAIS 서비스 계층**으로 서빙까지 떠안으면 Tier 2입니다. 그 선을 어디에 긋느냐가 7.2의 주제입니다.

## 7.2 책임 경계 — 2티어

GPUaaS의 가장 중요한 설계 결정은 **"PAIF가 어디까지 책임지고, 그 위는 테넌트 몫인가"** 입니다. 두 티어로 정리합니다.

| | Tier 1 — IaaS형 (GPU 자원 제공) | Tier 2 — PaaS형 (서빙까지 제공) |
|---|---|---|
| 한 줄 | GPU + 네임스페이스까지만 빌려줌 | 공용 모델 서빙 런타임까지 제공 |
| 테넌트가 받는 것 | GPU 바인딩된 VM 또는 VKS 네임스페이스 | 위 + 바로 쓰는 추론 엔드포인트 |
| 모델·서빙 | **테넌트가 직접**(자유 프레임워크) | **플랫폼이 운영**(테넌트는 모델만 올림) |
| 모델 라이프사이클 | 테넌트 책임 | 플랫폼 책임 |
| PAIS와의 관계 | 깔끔히 분리 | PAIS Model Runtime 영역과 중복 |
| 플랫폼팀 부담 | 낮음(자원·격리만) | 높음(모델 운영까지) |
| 적합 테넌트 | 데이터사이언티스트·MLOps(자체 스택) | 엔드포인트만 원하는 앱 개발자 |

### 경계선을 어디에 그을까 — 결정 가이드

핵심 판단 기준은 **"모델 라이프사이클을 누가 소유하나"** 입니다.

- **Tier 1에 선을 긋는다** — 테넌트가 자체 ML 스택·프레임워크를 갖고 있고 유연성을 원할 때, PAIF/PAIS 책임을 명확히 분리하고 싶을 때, 플랫폼팀이 모델별 운영 부담을 지지 않으려 할 때. **전통적 GPUaaS(IaaS)** 이며 가장 깔끔합니다.
- **Tier 2까지 확장한다** — 테넌트 다수가 "그냥 엔드포인트만 달라"는 앱 개발자이고, 표준화·활용률 극대화가 목표이며, 플랫폼팀이 공용 모델 운영을 떠안을 수 있을 때. 이 경우 사실상 **PAIS Model Endpoint를 서비스로 제공**하는 것이며, 그 위 RAG/에이전트는 [문서 04](04-dev-scenarios.md)·[문서 05](05-agents-mcp.md)로 이어집니다.

> 권장: **Tier 1을 기본 제공으로 깔고, Tier 2는 옵션 메뉴로** 올리는 혼합. 테넌트가 자원만 원하면 Tier 1, 서빙까지 원하면 Tier 2를 카탈로그에서 고르게 합니다. 한 플랫폼이 두 수준을 동시에 제공할 수 있습니다.

## 7.3 제공 형태 — VM + K8s 셀프서비스

PAIF는 GPU를 **두 가지 그릇**으로 테넌트에게 줄 수 있고, VCF의 강점은 둘을 한 플랫폼에서 셀프서비스로 제공하는 것입니다.

| 그릇 | 무엇을 받나 | 적합 |
|------|-------------|------|
| **vSphere VM** | GPU 바인딩된 VM(예: DLVM) | 데이터사이언티스트·연구·노트북 워크플로우 |
| **VKS 네임스페이스** | GPU 할당된 쿠버네티스 네임스페이스/클러스터 | 컨테이너·MLOps·CI/CD·확장형 추론 |

**셀프서비스 메커니즘:**

- **VM Class**: vGPU 또는 MIG 프로파일을 담은 커스텀 VM Class를 만들어 vSphere Namespace에 연결합니다. 테넌트는 카탈로그에서 이 클래스를 골라 GPU VM/노드를 받습니다.
- **VCF Automation 카탈로그**: 9.1에서 VCF Automation은 **GPU-enabled 리소스**를 셀프서비스 카탈로그/블루프린트로 제공합니다. 테넌트는 UI에서 GPU 자원을 신청하고, 승인·쿼터·정책이 자동 적용됩니다.
- **VKS 셀프서비스**: 개발팀이 네임스페이스·VKS 클러스터를 셀프서비스로 생성하고, 필요 시 DSM 관리형 DB까지 함께 받습니다([문서 03](03-workflows.md)).

> 즉 테넌트는 인프라팀에 티켓을 넣고 기다리지 않고, **카탈로그에서 GPU 자원을 직접 신청**합니다.

## 7.4 GPU 분할·할당 매트릭스

GPUaaS의 자원 효율은 GPU를 어떻게 나누느냐에서 갈립니다. 9.1은 세 방식을 제공하며, **NVAIE(NVIDIA AI Enterprise) 라이선스 필요 여부**가 비용에 직접 영향을 줍니다.

| 방식 | 분할 단위 | NVAIE | 특성 | 권장 용도 |
|------|-----------|-------|------|-----------|
| **Enhanced DirectPath I/O** | GPU 전체(전용 패스스루) | 불필요 | 테넌트당 전용 GPU, vMotion 이점 유지 | 대규모 학습·단일 워크로드 최대 성능·라이선스 절감 |
| **vGPU (타임슬라이스)** | 시간 분할 공유 | 필요 | 여러 테넌트가 한 GPU 시분할, 활용률↑ | 다수 소규모 추론·개발·노트북 |
| **MIG (Multi-Instance GPU)** | 하드웨어 분할 | 필요(vGPU 경유) | 한 GPU를 격리된 인스턴스로 쪼개 병렬 실행 | 강격리가 필요한 멀티테넌트 추론 |

**보조 개념:**

- **GPU Reservation**: vGPU 프로파일별로 GPU 용량을 **사전 예약**해, 예약된 용량이 특정 테넌트의 네임스페이스에서만 접근되도록 합니다. 멀티테넌트·계열사 공유에서 "옆 테넌트가 내 GPU를 잠식"하는 문제를 막는 핵심 장치입니다.
- **DRA(Dynamic Resource Allocation)**: 쿠버네티스에서 GPU를 동적으로 할당하는 방식([문서 02](02-architecture.md)·[문서 06](06-production.md) 참조).

**의사결정 지점:**

- **전용 vs 공유**: 활용률을 높이려면 vGPU/MIG 공유, 성능·라이선스 절감을 원하면 DirectPath 전용. GPUaaS는 보통 **공유(vGPU/MIG)가 기본**이고, 전용 풀을 별도 티어로 둡니다.
- **NVAIE 비용**: vGPU 분할·MIG·NIM/NeMo를 쓰면 NVAIE가 필요합니다([문서 01 §1.2](01-concepts.md)). DirectPath 전용만 쓰면 NVAIE가 불필요하므로, "전용 GPU 풀"은 라이선스 비용 측면에서 유리할 수 있습니다.
- **MIG 지원 GPU 확인**: MIG는 지원 데이터센터 GPU(A100·A30·H100 등)에서만 동작합니다. 모델별 지원 여부는 **공식 Compatibility Guide(BCG/HCL)** 로 반드시 확인하세요. GPU Operator 25.10.1(드라이버 v580.x) 기준 MIG 설정은 [문서 00](00-whats-new.md)의 재검증 항목을 참고합니다.

## 7.5 멀티테넌시·격리·쿼터

GPUaaS는 본질적으로 멀티테넌트입니다. 격리 수준은 테넌트가 누구냐(같은 조직 팀 vs 다른 계열사)에 따라 달라집니다.

| 격리 축 | 메커니즘 |
|---------|----------|
| 자원 격리 | 네임스페이스 + 리소스 쿼터(GPU 개수·메모리·CPU), GPU Reservation으로 용량 보장 |
| 네트워크 격리 | NSX 마이크로세그멘테이션, 네임스페이스/테넌트 간 트래픽 차단([문서 06](06-production.md)) |
| 데이터 격리 | 테넌트별 스토리지 정책, DSM Data Service Policy, 접근 통제 |
| 권한 격리 | OIDC/RBAC, 조직(Organization)·프로젝트 단위 권한 |

**격리 강도 — 단일 조직 vs 계열사:**

- **단일 조직 내 팀들**: 네임스페이스 + 쿼터 + RBAC로 충분한 경우가 많습니다. 신뢰 경계가 같은 회사 안이라 상대적으로 가볍습니다.
- **계열사 간 공유 풀**: 법인이 다르므로 **강격리**가 필요합니다 — GPU Reservation으로 용량을 법인별로 못박고, NSX로 네트워크를 분리하며, MIG로 하드웨어 수준 격리를 더하고, 데이터·권한을 조직 단위로 끊습니다. 회귀 테스트에 **"타 계열사 자원·데이터 침범 불가"** 케이스를 반드시 포함합니다.

## 7.6 측정·과금 — 쇼백에서 차지백으로

GPUaaS가 "서비스"가 되려면 **누가 얼마나 썼는지**가 보여야 하고, 그룹으로 확장되면 **정산**까지 필요합니다. 성숙도에 따라 단계적으로 올립니다.

> **비용·TCO 산정과 용량 계획의 기준 문서는 [⑥ 사이징·용량·비용 가이드](../../06-sizing-cost/README.md)입니다.** 본 절은 GPUaaS 맥락의 과금(쇼백·차지백) 모델 개요이며, 상세 비용 워크시트·퍼블릭 대비 비교·용량 계획은 ⑥을 참조하시기 바랍니다.

**1단계 — 쇼백(가시화):** 테넌트별 사용량과 비용을 **보여주되 청구는 하지 않습니다.** VCF Operations의 **Application Showback**으로 테넌트/애플리케이션 단위 비용을 가시화합니다. 목적은 투명성과 낭비 감소 — 사용량이 보이면 팀 스스로 최적화를 유도할 수 있습니다.

**2단계 — 차지백(정산):** 사전 정의된 가격 정책으로 테넌트에게 **실제 비용을 청구**합니다. VCF Operations는 **Cost Drivers(비용 귀속)·Pricing Policies(요율표)·Chargeback**을 제공하며, 9.1에서는 **VKS 노드 차지백**과 **배포 전 실시간 견적**(카탈로그 신청 시 비용 미리 표시)이 추가됐습니다.

**GPU 비용 귀속 — 정직한 한계:** VCF Operations의 Rate Card는 컴퓨트(CPU·메모리)·스토리지·네트워크·태그·고정비 등을 다루지만, **GPU 단위(vGPU/시간)의 네이티브 미터링은 공식 문서에 명시적으로 보장돼 있지 않습니다.** 따라서 GPU 비용은 다음으로 **모델링**하는 접근이 현실적입니다.

- **태그 기반**: GPU VM/네임스페이스에 테넌트·GPU 등급 태그를 달고, 태그를 Cost Driver로 귀속
- **VM Class 기반**: vGPU 프로파일별 VM Class에 고정 단가를 매핑
- **전용 풀 기반**: GPU 호스트/클러스터를 테넌트 전용 풀로 잡고 풀 단위 비용을 배분

> 적용 전 **GPU 미터링을 어느 단위로 잡을지(점유 시간 vs 할당 용량 vs 전용 풀)** 와 VCF Operations Rate Card 연동 방식을 공식 문서·PoC로 검증하세요. 본 절은 모델링 방향을 제시할 뿐, 특정 미터링 항목의 존재를 단정하지 않습니다.

## 7.7 관측성·Day-2 운영

GPUaaS 운영자는 "테넌트 앱이 잘 도나"가 아니라 **"GPU 자원이 잘 쓰이나"** 를 봅니다.

- **GPU 활용률 관측**: 9.1은 모델·**GPU 대시보드**를 기본 제공합니다([문서 06 §관측성](06-production.md)). 테넌트별·GPU별 활용률, 유휴 GPU, 메모리 사용을 모니터링해 풀 효율을 관리합니다.
- **테넌트별 지표**: 쿼터 소진율, 대기/거부된 GPU 요청, 테넌트별 점유 시간 — 용량 계획과 차지백의 입력값입니다.
- **드라이버·펌웨어 라이프사이클**: GPU 드라이버(v580.x)·GPU Operator(25.10.1)·MIG 설정의 버전 관리와 회귀 테스트([문서 00](00-whats-new.md)). 드라이버 교체는 테넌트 워크로드에 영향을 주므로 유지보수 창과 롤백 계획이 필요합니다.
- **용량 계획**: 유휴율·대기 큐를 보고 GPU 증설 시점을 판단. GPU Reservation 예약분과 가용분의 균형을 관리합니다.

**Day-2 체크리스트:**

```
- GPU 활용률·유휴율 대시보드 상시 모니터
- 테넌트 쿼터 소진·거부 요청 추적
- 드라이버/GPU Operator/MIG 버전 관리 + 교체 시 회귀 테스트
- GPU Reservation 예약분 vs 가용분 균형 점검
- 멀티테넌트 격리 회귀(타 테넌트 자원·데이터 침범 불가)
- (그룹) 차지백 정산 데이터 일관성 검증
```

## 7.8 레퍼런스 시나리오

### 시나리오 A — 사내 GPU 셀프서비스 플랫폼 (단일 조직, 쇼백)

**상황:** 한 기업의 여러 데이터사이언스·개발팀이 GPU를 필요로 하나, 팀마다 GPU 서버를 따로 사서 활용률이 낮고 관리가 분산돼 있다. 중앙 플랫폼팀이 GPU를 공용 풀로 모아 셀프서비스로 제공하려 한다.

**구성:**
- **티어**: Tier 1(IaaS) 기본 — 팀은 GPU 자원을 받아 자체 프레임워크로 모델을 돌린다. 표준 추론이 필요한 팀에는 Tier 2(공용 서빙) 옵션 제공.
- **제공 형태**: 데이터사이언티스트는 GPU VM(노트북형), MLOps팀은 VKS 네임스페이스. 둘 다 VCF Automation 카탈로그에서 셀프 신청.
- **분할**: 개발·실험은 vGPU 타임슬라이스로 활용률 극대화, 대규모 학습은 DirectPath 전용 풀.
- **격리**: 네임스페이스 + 쿼터 + RBAC(같은 회사라 경량).
- **과금**: 쇼백 — 팀별 사용량·비용 가시화로 자율 절감 유도(청구는 안 함).

**기대 효과:** GPU 활용률 향상(공용 풀링), 신청-대기 제거(셀프서비스), 분산 관리 → 중앙 표준화.

### 시나리오 B — 계열사 공유 GPU 풀 (그룹, 차지백, 강격리)

**상황:** 지주사 산하 여러 계열사가 각자 GPU를 사기엔 비효율적이라, 그룹 차원의 공용 GPU 풀을 만들어 계열사에 나눠 쓰게 하려 한다. 법인이 다르므로 자원·비용·데이터가 엄격히 분리·정산돼야 한다.

**구성:**
- **티어**: Tier 1 + Tier 2 혼합 — 계열사별 성숙도에 따라 자원만(Tier 1) 또는 서빙까지(Tier 2) 선택.
- **제공 형태**: 계열사를 조직(Organization) 단위 테넌트로, VCF Automation에서 계열사별 카탈로그·쿼터.
- **분할·격리(강)**: **GPU Reservation**으로 계열사별 GPU 용량 못박기 + **MIG** 하드웨어 격리 + **NSX** 네트워크 분리 + 조직 단위 데이터·권한 격리.
- **과금**: **차지백** — 계열사별 실제 사용량을 정산. 태그/VM Class/전용 풀 기반으로 GPU 비용을 귀속(7.6의 한계·검증 전제). 배포 전 실시간 견적으로 계열사가 비용을 미리 확인.

**기대 효과:** 그룹 GPU 투자 효율 극대화, 계열사 간 공정한 비용 정산, 법인 경계 강격리로 규제·보안 충족.

> 두 시나리오는 별개가 아니라 **성숙도 진행**입니다. 사내 셀프서비스(A)로 시작해 그룹 공유(B)로 확장하면, 쇼백→차지백·경량격리→강격리·단일조직→다법인으로 자연스럽게 올라갑니다.

## 7.9 성숙도 로드맵

[문서 04 §4.3 성숙도 축](04-dev-scenarios.md)을 GPUaaS 관점으로 구체화하면 다음과 같습니다.

```
1단계: 단일팀 풀링
  - 한 팀/소수 팀이 공용 GPU 풀 사용, vGPU 공유, 네임스페이스 쿼터
  - 과금 없음(또는 단순 사용량 로깅)
        ↓
2단계: 부서 셀프서비스 (시나리오 A)
  - 다수 팀, VCF Automation 카탈로그 셀프서비스, VM+VKS 병행
  - 쇼백(사용량·비용 가시화), 경량 격리(네임스페이스+RBAC)
        ↓
3단계: 그룹 공유 풀 (시나리오 B)
  - 계열사 멀티테넌트, GPU Reservation+MIG+NSX 강격리
  - 차지백 정산, 배포 전 견적, 거버넌스·감사
```

**도입 시 핵심 체크:**

```
- 경계선 결정: Tier 1만 / Tier 1+2 (모델 라이프사이클 소유자)
- 분할 전략: vGPU 공유 vs DirectPath 전용 풀, NVAIE 비용 산정
- MIG 지원 GPU를 BCG/HCL로 확인
- 격리 강도: 단일조직(경량) vs 계열사(강격리: Reservation+MIG+NSX)
- 과금 단계: 쇼백 먼저, 차지백은 GPU 미터링 단위 검증 후
- 셀프서비스 카탈로그(VCF Automation) + 쿼터·승인 정책 설계
- 관측성: GPU 활용률·테넌트 지표 대시보드, 드라이버 라이프사이클
```
