# 07 — 산업 도입 시나리오

> 금융/유통/제조 산업별 VCF DSM + pgvector 적용 시나리오

기준 버전: VCF 9.1 / DSM 9.1 / PAIS 2.1. 아키텍처 배경은 [03 아키텍처](03-vcf-dsm-architecture.md), 배포 절차는 [04 배포](04-deployment.md)를 참조하시기 바랍니다.

---

## 7.1 시나리오 A: 금융권 — 사내 규정 검색 AI 어시스턴트

**상황**: 국내 대형 은행. 내부 규정, 지침, 매뉴얼이 수만 건에 달하며, 신규 직원이나 지점 상담원이 관련 규정을 찾는 데 상당한 시간이 소요됩니다. 기존 키워드 검색은 정확한 용어를 알아야 결과가 나오며, 유사 규정 간 교차 검색이 어렵습니다.

**요구사항**
- 자연어로 질문하면 관련 규정을 찾아 답변하는 AI 어시스턴트
- 모든 데이터가 은행 내부 데이터센터에 위치해야 함 (금감원 규제)
- 기존 VCF 인프라 활용, 추가 인프라 구매 최소화
- 규정 변경 시 자동으로 지식 기반 업데이트

**VCF + DSM + PAIS 기반 솔루션 구성**

인프라 레이어에서는 VCF Workload Domain(기존 인프라)을 활용하고, vSAN ESA에 FTT=2(RAID-6)를 적용하여 금융권 수준의 데이터 보호를 제공하며, NSX 마이크로세그멘테이션으로 AI 워크로드 네트워크를 격리합니다.

데이터 레이어에서는 DSM으로 PostgreSQL + pgvector HA 클러스터(3노드)를 프로비저닝하고, Cross-Cluster HA로 서로 다른 vSphere Cluster에 노드를 분산 배치합니다. 자동 백업 + PITR(Point-In-Time Recovery, 특정 시점 복구)을 활성화하여 RPO(Recovery Point Objective, 목표 복구 시점)를 분 단위로 유지합니다.

AI 레이어에서는 PAIS Model Runtime에 임베딩 모델(예: 다국어 모델 bge-m3 또는 한국어 대응 가능한 multilingual-e5-large)과 LLM(예: Llama 3.1 70B 또는 한국어 특화 모델)을 vLLM/Infinity 기반으로 배포합니다. PAIS Data Indexing & Retrieval로 규정 문서 소스(SharePoint, 파일 서버)를 연결하고, 청킹 → 임베딩 → pgvector 저장을 자동화합니다. 규정 변경 감지 시 자동 재인덱싱을 스케줄링합니다. 데이터 반출이 불가한 경우 PAIS 2.1의 폐쇄망(에어갭, Artifact Mirroring Tool) 구성으로 외부 연결 없이 운영합니다.

애플리케이션 레이어에서는 PAIS Agent Builder에서 "규정 검색 에이전트"를 구성하고, 기존 인트라넷 포털에 OpenAI 호환 API로 연동합니다.

**기대 효과**
- 규정 검색 시간: 평균 15–30분 → 수 초
- 추가 인프라 비용: VCF 기존 인프라 활용으로 최소화. GPU 노드 및 NVIDIA AI Enterprise 라이선스 추가 필요 (온프레미스 LLM/임베딩 모델 운영 시)
- 데이터 주권: 모든 데이터가 은행 데이터센터 내에 위치
- 운영 부담: DSM 자동화로 DBA 개입 최소화

**예상 규모**: 규정 문서 약 5만 건 → chunk 기준 약 200만–500만 벡터 → DSM 단일 HA 클러스터로 충분

## 7.2 시나리오 B: 유통/리테일 — 상품 추천 및 고객 상담 AI

**상황**: 대형 유통 그룹. 수백만 SKU의 상품 카탈로그를 보유하고 있으며, 온라인몰과 오프라인 매장의 고객 상담 품질을 AI로 개선하고자 합니다. 기존 키워드 기반 상품 검색은 "여름에 입기 좋은 시원한 소재 원피스"같은 자연어 검색을 처리하지 못합니다.

**요구사항**
- 자연어 상품 검색: "캠핑 가서 바비큐 할 때 필요한 것들" → 관련 상품 추천
- 고객 상담 AI: 상품 문의, 반품/교환 정책, 매장 안내 등에 대한 자동 응답
- 기존 ERP/POS 데이터와 연동
- 피크 시즌(명절, 세일) 대응 가능한 성능

**VCF + DSM 기반 솔루션 구성**

상품 벡터 DB로 DSM PostgreSQL + pgvector를 활용합니다. 상품명, 설명, 카테고리, 속성 텍스트를 임베딩하여 벡터로 저장하고, 상품 가격, 재고, 카테고리 등 관계형 데이터와 벡터가 같은 DB에 위치하므로 "가격 5만원 이하 + '가벼운 여름 원피스' 의미 검색"을 단일 SQL로 처리합니다. pgvector 0.8.0의 Iterative Index Scan이 카테고리 필터 + 벡터 검색을 효율적으로 결합합니다.

고객 상담 지식 기반으로는 PAIS Data Indexing & Retrieval이 상품 FAQ, 반품/교환 정책, 매장 안내 문서를 인덱싱합니다. Agent Builder에서 "상담 에이전트"를 구성하여 채팅 인터페이스로 제공합니다.

**기대 효과**
- 상품 검색 정확도: 키워드 매칭 대비 의미 검색으로 검색 만족도 향상
- 고객 상담 응답 시간: 대기 시간 제거, 24/7 자동 응답
- 동기화 오버헤드 제거: 상품 데이터 변경이 별도 벡터 DB 동기화 없이 즉시 반영
- 피크 대응: DSM Read Replica 추가 + vSphere DRS 기반 리소스 자동 조정

**예상 규모**: SKU 300만 건 + 상담 FAQ 1만 건 → 약 500만–800만 벡터 → DSM HA 클러스터 + Read Replica 1–2개로 충분

## 7.3 시나리오 C: 제조/엔지니어링 — 기술 문서 검색 및 이상 탐지

**상황**: 대형 제조 그룹. 수십 년간 축적된 설비 매뉴얼, 유지보수 이력, 장애 대응 기록이 있으나, 신규 엔지니어가 이 지식에 접근하기 어렵습니다. 동시에 IoT 센서 데이터 기반 설비 이상 탐지를 AI로 고도화하고자 합니다.

**VCF + DSM 기반 솔루션 구성**

기술 문서 검색은 설비 매뉴얼, 유지보수 기록, 장애 대응 보고서를 pgvector에 임베딩하여 저장합니다. "펌프 진동 증가 시 점검 절차"같은 자연어 검색으로 관련 문서를 즉시 찾습니다.

이상 탐지는 정상 운전 상태의 센서 패턴을 벡터로 학습하여 pgvector에 저장하고, 실시간 센서 데이터를 벡터화하여 정상 패턴과의 거리를 측정하여, 임계값 이상 이탈 시 이상 알림을 발생시킵니다.

두 워크로드가 같은 DSM PostgreSQL + pgvector 인스턴스에서 동작하되, 스키마와 인덱스로 분리하여 관리합니다. 제조 환경 특성상 데이터가 외부로 나갈 수 없으므로 온프레미스 VCF 구성이 필수적이며, 방산 등 보안 등급이 높은 환경은 PAIS 2.1 폐쇄망(Artifact Mirroring Tool) 구성을 적용합니다.
