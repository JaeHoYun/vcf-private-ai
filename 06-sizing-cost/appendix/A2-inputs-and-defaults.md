# 부록 A2 — 입력값 환산·기본값·모델 선택 보조

> 기반 버전은 [README 버전 기준 문서](../README.md#기반-버전-source-of-truth)를 참조하세요.
> 시리즈 인덱스: [vcf-private-ai-series](../../README.md)

본문 [01.3 사이징 입력값 체크리스트](../docs/01-sizing-methodology.md#13-사이징-입력값-체크리스트)는 "무엇을 채워야 하는가"를 알려 주지만, 처음 사이징하는 사람은 **그 칸을 어떤 숫자로 채울지**부터 막힙니다. 이 부록은 "임직원 N명"·"사내 챗봇" 같은 **사업 언어를 사이징 입력(동시성·QPS[초당 질의 수]·컨텍스트·SLA)으로 환산**하는 보조표와, 모델 선택의 1차 가이드를 제공합니다.

> 여기의 모든 기본값(계수)은 **출발 가정치**입니다. 조직의 실제 사용 패턴으로 보정하고, 확정값은 [06.5 파일럿 실측](../docs/06-capacity-planning.md#65-poc--파일럿--프로덕션-용량-로드맵)으로 갈음하세요.

## A2.1 사업 언어 → 사이징 입력 환산

가장 자주 막히는 변환은 "**사용자 수 → 동시 요청 수**(concurrency)"입니다. GPU 사이징을 좌우하는 것은 전체 사용자 수가 아니라 **피크 순간의 동시 처리 중 요청 수**입니다.

### 환산 사슬 (5단계)

| 단계 | 입력 | 기본 가정(보정 대상) | 산식 |
|---|---|---|---|
| 1 전체 사용자 | 임직원/대상자 수 N | — | N |
| 2 일 활성(DAU) | 활성 비율 d | 사내 도구 d ≈ 20–40% | DAU = N × d |
| 3 일 요청 수 | 1인당 일 질의 q | 보조 도구 q ≈ 5–20 | 일요청 = DAU × q |
| 4 피크 QPS | 업무시간 H, 피크/평균비 p | H ≈ 8h, p ≈ 3 | 피크QPS = 일요청 ÷ (H×3600) × p |
| 5 동시 요청 | 요청당 처리시간 t(초) | RAG t ≈ 3–6초 | 동시성 ≈ 피크QPS × t |

> 동시성 ≈ 피크QPS × 평균 요청 지속시간(리틀의 법칙). LLM 요청은 토큰을 스트리밍하는 동안 GPU 슬롯을 점유하므로, "초당 요청"보다 "**동시에 떠 있는 요청 수**"가 메모리·GPU 점유의 직접 동인입니다.

### 동시성에서 다시 헤드룸 가산

산출된 동시성은 평균적 피크입니다. 여기에 버스트·가용성 여유를 더합니다.

- 버스트 헤드룸: +20–30% (본문 [01.5](../docs/01-sizing-methodology.md#15-핵심-원칙-추정으로-시작하되-실측으로-보정))
- 가용성(N+1): 노드/Replica 1단위 추가 (본문 [03.6](../docs/03-compute-memory-sizing.md#36-replica동시성-환산-워크로드--pod-replica--노드-수요))

## A2.2 워크로드 프로파일 프리셋

대표 워크로드별 입력 기본값입니다. 자사 값이 없을 때 출발점으로 쓰고, 확정 전 보정하세요.

| 프로파일 | 입력 토큰(어림) | 출력 토큰(어림) | 동시성 경향 | 지배 자원 | 메모 |
|---|---|---|---|---|---|
| 사내 Q&A 챗봇 | 0.5–2K | 0.2–0.5K | 낮음–중간 | 추론 GPU | 컨텍스트 짧음, KV 부담 작음 |
| 사내문서 RAG | 2–8K(검색 청크 포함) | 0.3–1K | 중간 | 추론 GPU + 벡터DB | 입력 길어 KV·임베딩 추가 |
| 코드 어시스턴트 | 2–16K | 0.5–2K | 중간 | 추론 GPU(장문) | 긴 컨텍스트로 KV 압박 |
| 문서 요약(배치) | 8–32K | 0.5–2K | 버스트성 | 추론 GPU(장문) | 야간 배치, 저지연 불필요 |
| 임베딩 인덱싱(배치) | — | — | 버스트성 | CPU/중급 GPU | LLM과 분리 권장 |
| 멀티모델 서빙(부서별) | 모델별 상이 | 모델별 상이 | 중간 | 추론 GPU(합산/MIG) | 모델 수만큼 VRAM 합산, MIG 격리(본문 [02 §2.9](../docs/02-gpu-sizing.md#29-고정-gpu-적재패킹-공급-제약)) |
| 파인튜닝(LoRA/QLoRA) | 학습셋 의존 | — | 배치성·간헐 | GPU 메모리(어댑터·옵티마이저) | 야간·선점형, 추론과 노드 분리(본문 [02 §2.10](../docs/02-gpu-sizing.md#210-학습파인튜닝-gpu-메모리)) |
| 풀 파인튜닝·사전학습 | 학습셋 의존 | — | 버스트성·대형 | 다중 GPU(샤딩) | FSDP/ZeRO-3, 노드 인터커넥트 중요(본문 [02 §2.10](../docs/02-gpu-sizing.md#210-학습파인튜닝-gpu-메모리)) |

> 멀티모달(이미지·오디오·문서 업로드)이 섞이면 입력 바이트·네트워크·메모리가 급증합니다. 워크로드 믹스를 먼저 고정하세요(본문 [05.4](../docs/05-storage-network-sizing.md#54-네트워크-용량--추론반입내부-통신)).
> 학습·파인튜닝 프리셋은 추론과 메모리 셈법이 다릅니다(가중치 외 그래디언트·옵티마이저·활성화). 파라미터당 메모리와 LoRA/QLoRA·샤딩은 본문 [02 §2.10](../docs/02-gpu-sizing.md#210-학습파인튜닝-gpu-메모리)을 참조하세요.

## A2.3 지연·가용성 SLA 기본값

확정 SLA가 없을 때 출발점으로 쓰는 보수적 기본값입니다(조직 합의로 확정).

| 지표 | 정의 | 기본값(출발) | 근거·메모 |
|---|---|---|---|
| TTFT | 첫 토큰까지 시간 | 1–2초(대화형 상한 200ms~) | VMware는 200ms 목표 제시([출처](https://blogs.vmware.com/cloud-foundation/2024/09/25/llm-inference-sizing-and-performance-guidance/)); 사내 도구는 다소 완화 가능 |
| TPOT/ITL | 토큰 간 지연 | 사람 읽기 속도(약 10–20 tok/s) 이상 | 체감 생성 속도 |
| P95/P99 | 백분위 지연 | SLA는 평균이 아닌 P95/P99로 | 본문 [01.3](../docs/01-sizing-methodology.md#13-사이징-입력값-체크리스트) |
| 가용성 | 다중화 수준 | 최소 N+1 | 노드 장애·롤링 업데이트 흡수 |
| 헤드룸 | 평균 사용률 여유 | 20–30% | 본문 [06.2](../docs/06-capacity-planning.md#62-증설-트리거--임계헤드룸버스트-정책) |

## A2.4 모델 선택 1차 가이드

"어떤 모델을 몇 B로 돌릴 것인가"는 사이징의 출발 입력입니다. 본 시리즈는 모델 제품을 단정하지 않으나, **규모 선택의 사고 틀**은 다음과 같습니다(아키텍처·운영 상세는 ① [vcf-private-ai-guide](../../01-infra/README.md), 서빙은 ③ [vcf-paif-serving-api-guide](../../03-serving-api/README.md) 참조).

| 판단 축 | 작은 모델(약 7–13B) | 중형(약 30–34B) | 대형(약 70B+) |
|---|---|---|---|
| 단순 Q&A·분류·요약 | 대개 충분 | 과할 수 있음 | 불필요 경향 |
| 사내문서 RAG | 충분한 경우 많음(지식은 벡터DB) | 품질 여유 | 복잡 추론 필요 시 |
| 복잡 추론·에이전트 | 부족할 수 있음 | 균형점 | 유리 |
| VRAM(FP16 가중치) | 약 14–26GB | 약 60–68GB | 약 140GB+ |
| 단일 80GB GPU 적재 | 여유 | 양자화 시 가능 | 불가(멀티 GPU/양자화 필수) |

선택 원칙:

- **RAG는 모델 규모를 키우는 1순위 수단이 아닙니다.** 지식은 모델 밖 벡터DB에 있으므로(본문 [01.2](../docs/01-sizing-methodology.md#12-워크로드-분류와-자원-특성)), 검색 품질·청킹·리랭킹 개선이 모델 키우기보다 비용 효율적인 경우가 많습니다.
- **작게 시작해 PoC로 올린다.** 가장 작은 후보로 태스크 적합성을 먼저 검증하고, 품질이 부족할 때만 키웁니다([06.5](../docs/06-capacity-planning.md#65-poc--파일럿--프로덕션-용량-로드맵)).
- **양자화를 규모 선택과 함께 본다.** INT8/FP8은 통상 품질 영향이 작고 메모리를 약 절반으로 줄입니다(본문 [02.4](../docs/02-gpu-sizing.md#24-양자화-메모리와-품질의-트레이드오프)). 대형 모델을 양자화해 중형 비용으로 운영하는 선택지가 있습니다.
- **한국어 등 언어 품질은 파라미터 수만으로 정해지지 않습니다.** 후보 모델을 자사 태스크 데이터로 평가해 확정하세요(실측 필요).

## A2.5 모델 구조값 확인 방법 (KV 캐시 계산 입력)

본문 [02.3](../docs/02-gpu-sizing.md#23-kv-캐시-산정-컨텍스트--동시성)의 KV 캐시 산식에는 모델 구조값이 필요합니다. 출처는 모델 카드와 함께 배포되는 `config.json`입니다.

| 산식 항목 | `config.json` 키(예시) | 비고 |
|---|---|---|
| 레이어 수 | `num_hidden_layers` | 트랜스포머 블록 수 |
| 어텐션 헤드 수 | `num_attention_heads` | — |
| KV 헤드 수 | `num_key_value_heads` | **어텐션 헤드보다 작으면 GQA** → KV 캐시 절감 |
| head_dim | `hidden_size ÷ num_attention_heads` 또는 `head_dim` | 직접 명시되기도 함 |
| 정밀도 | `torch_dtype` | FP16/BF16=2바이트 등 |

> 판별 팁: `num_key_value_heads < num_attention_heads`이면 GQA(Grouped Query Attention)이며, 같은 파라미터라도 KV 캐시가 크게 줄어 동시성에 유리합니다(본문 [02.3](../docs/02-gpu-sizing.md#23-kv-캐시-산정-컨텍스트--동시성)). 구조값을 [부록 A1](A1-first-order-reference.md)의 토큰당 KV 계수와 교차 확인하세요.

## A2.6 입력값 워크시트 (복사해서 채우기)

아래를 채우면 본문 02부터 바로 진행할 수 있습니다. 모르는 칸은 "확인 필요"로 두고 A1·A2 기본값으로 출발하세요.

```
[워크로드]
- 용도/프로파일:            (예: 사내문서 RAG)
- 대상 사용자 수 N:
- DAU 비율 d:               (기본 20–40%)
- 1인당 일 질의 q:          (기본 5–20)
- 피크/평균비 p:            (기본 3)
- 요청당 처리시간 t:        (RAG 기본 3–6초)
→ 피크 동시성(계산):
→ +버스트/N+1 헤드룸:

[모델]
- 후보 모델·파라미터 수:
- 정밀도(FP16/FP8/INT8):
- 구조값(layers/KV heads/head_dim): (config.json)

[컨텍스트]
- 입력 토큰(평균/최대):
- 출력 토큰(평균):

[SLA]
- TTFT 목표 / TPOT / P95:
- 가용성(N+1 등):
```

> 이 워크시트가 채워지면 [08 — 레퍼런스 시나리오](../docs/08-reference-scenario.md)의 흐름을 그대로 따라 GPU→노드→클러스터→스토리지→TCO까지 환산할 수 있습니다.
