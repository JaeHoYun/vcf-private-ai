# 08 — 레퍼런스 시나리오 (전 과정 예제: 입력 → GPU → 노드 → 클러스터 → 스토리지 → TCO)

> 기반 버전은 [README 버전 기준 문서](../README.md#기반-버전-source-of-truth)를 참조하세요.
> 시리즈 인덱스: [vcf-private-ai-series](../../README.md)

이 문서는 01–07의 산식을 **하나의 시나리오로 처음부터 끝까지** 실행해 보이는 워크북입니다. 흩어진 예시 숫자가 아니라, 동일한 가정이 입력에서 비용까지 끊김 없이 이어지도록 구성했습니다. 각 단계의 숫자 옆에는 근거가 된 본문 절을 링크해 **추적 가능**하게 둡니다.

> **중요 — 이 예제의 모든 숫자는 "예산 추정용 1차 값"입니다.** 처리량·동시성 등 핵심 수치는 [부록 A1](../appendix/A1-first-order-reference.md) 가정치를 사용했으며, 확정 사이징은 [06.5 PoC→파일럿→프로덕션](06-capacity-planning.md#65-poc--파일럿--프로덕션-용량-로드맵) 실측으로 갈음해야 합니다. 금액(단가)은 본 시리즈 원칙에 따라 제시하지 않으며 견적으로 채웁니다.

## 8.0 시나리오 정의

| 항목 | 값 |
|---|---|
| 용도 | 사내 임직원 대상 **한국어 사내문서 RAG 챗봇**(규정·기술문서 Q&A) |
| 대상 사용자 | 임직원 약 **5,000명** |
| 배포 | 온프레미스(데이터 주권), VCF 9.1 / PAIF 9.1 / PAIS 2.1, VKS |
| 단계 | 구매 전 예산 산정 → 이후 PoC로 실측 |
| 워크로드 분류 | RAG = 추론 + 임베딩 + 벡터 검색 (본문 [01.2](01-sizing-methodology.md#12-워크로드-분류와-자원-특성)) |

이 시나리오의 결론을 먼저 말하면: **GPU 수요는 의외로 작고(수 장 규모), 노드 수와 비용은 플랫폼 최소요건·HA 정책이 좌우합니다.** RAG는 지식을 모델 밖 벡터DB에 두기 때문입니다. 아래에서 그 과정을 단계별로 보입니다.

## 8.2 단계 2 — GPU 메모리(VRAM) 산정 (본문 02 적용)

### 가중치 메모리 ([02.2](02-gpu-sizing.md#22-가중치-메모리-산정-파라미터--정밀도))

8B × 2바이트(FP16) ≈ **약 16GB**.

### KV 캐시 ([02.3](02-gpu-sizing.md#23-kv-캐시-산정-컨텍스트--동시성))

토큰당 KV = 2 × 32(레이어) × 8(KV헤드) × 128(head_dim) × 2(FP16) = 131,072 바이트 ≈ **0.000122 GiB/token**.
→ [부록 A1](../appendix/A1-first-order-reference.md#a11-추론-처리량동시성-1차-가정치)의 VMware 계산기 값(8B = 0.000122 GiB/token)과 **정확히 일치**(두 방법 교차검증).

| 항목 | 계산 | 결과 |
|---|---|---|
| 요청당 KV | 0.000122 × 4,096 | 약 0.5 GiB |
| 총 KV(동시성 21) | 0.5 × 21 | 약 10.5 GiB |

### 합산 (단일 GPU 적재 가능성)

| 구성요소 | 값 |
|---|---|
| 가중치 | 약 16 GiB |
| KV 캐시(동시성 21) | 약 10.5 GiB |
| 활성화·오버헤드(어림) | 약 3–5 GiB |
| **소계** | **약 30–32 GiB** |

`gpu_memory_utilization` 0.9 기준 80GB GPU의 가용 약 72GB([02.1](02-gpu-sizing.md#21-vram-산정의-3대-구성요소)) → **단일 80GB GPU 1장에 8B 모델 + 동시성 21(4K 컨텍스트)이 여유 있게 적재**됩니다.

> 만약 70B FP16였다면 가중치만 약 140GB로 단일 GPU 불가 → 멀티 GPU(텐서 병렬) 또는 양자화 필수([02.2·02.6](02-gpu-sizing.md#26-다중-gpu-텐서-병렬--파이프라인-병렬-개요)). 모델 규모 선택이 이후 모든 단계를 좌우함을 보여줍니다.

## 8.3 단계 3 — 처리량 → GPU 수 (본문 02.5 + 부록 A1)

"메모리에 들어가는가"는 8.2에서 확인했고, 이제 "목표 처리량을 내는가"입니다([02.5](02-gpu-sizing.md#25-처리량지연-목표--gpu-수replica-환산)).

| 점검 | 1차 가정치 | 출처 |
|---|---|---|
| 단일 80GB GPU 8B 처리량 | 단일 H100 측정 약 5,500–6,300 tok/s(짧은 입력 기준); 4K 입력은 더 낮음 | [A1.1](../appendix/A1-first-order-reference.md#a11-추론-처리량동시성-1차-가정치)(databasemart 측정) |
| 단일 GPU 8B 동시성(4K) | 64 이상(L40 48GB 기준) | A1.1(VMware) |
| 필요 출력 처리량(피크) | 3.1 QPS × 약 500 tok ≈ 약 1,550 tok/s | 8.1 입력 |

설계 동시성 21, 필요 처리량 약 1,550 tok/s는 **단일 80GB GPU 1장**(1 Replica)이 받칠 수 있는 범위입니다(A1 가정치 기준; 단일 H100 측정 처리량은 수천 tok/s). 여기에 가용성을 더합니다.

| 항목 | 수량 | 근거 |
|---|---|---|
| 추론 Replica | 1 (피크 충족) | 8.2·8.3 |
| 가용성 N+1 | +1 → **추론 GPU 2장** | [03.6](03-compute-memory-sizing.md#36-replica동시성-환산-워크로드--pod-replica--노드-수요) |
| 임베딩·리랭커 | 중급 GPU 1장(또는 MIG 공유) | [03.5](03-compute-memory-sizing.md#35-임베딩리랭커-서버-사이징rag-비-llm-컴포넌트)·[A1.2](../appendix/A1-first-order-reference.md#a12-임베딩리랭커-1차-가정치-rag-비-llm) |
| **총 GPU 수요(논리)** | **약 3 GPU** | 합산 |

> 핵심: 5,000명 RAG 챗봇의 GPU 수요는 약 3장입니다. 추론 부하 자체가 작고, 가용성·보조 컴포넌트가 수를 결정합니다. 이 수치는 부록 A1의 1차 가정치 기준이며, PoC 실측으로 대체할 대상입니다.

## 8.4 단계 4 — 노드·클러스터 구성 (본문 04 적용)

[04.6 사이징 워크시트](04-vks-cluster-sizing.md#46-인프라-고려사항과-사이징-워크시트)에 대입합니다.

| 단계 | 입력 | 결과 |
|---|---|---|
| 총 GPU 수요 | 8.3 결과 | 3 GPU |
| GPU 노드 사양 | 노드당 1 GPU(many-small, 추론 서빙) | [04.3](04-vks-cluster-sizing.md#43-노드-사양-산정-노드-크기-vs-노드-수) |
| GPU 워커 노드 수 | 3 ÷ 1 | 3 |
| **PAIF 최소 호스트 규칙** | 초기 클러스터 최소 3 GPU 호스트 | [01.1](01-sizing-methodology.md#11-사이징의-목적과-본-가이드의-위치)·[03.2](03-compute-memory-sizing.md#32-numapcie-배치-고려) |
| 일반 워커 노드 | 시스템 애드온·임베딩 CPU 경로 | 3 |
| 컨트롤 플레인 | HA(홀수) | 3 |
| **클러스터 노드 합계** | 3 + 3 + 3 | **9 노드(VM)** |
| Supervisor 한도 점검 | 9 노드 ≪ 4,000, 1 클러스터 ≪ 500 | 여유([04.5](04-vks-cluster-sizing.md#45-스케일-한도와-단일-대형-vs-다중-클러스터-설계)) |

이 시나리오의 교훈: **워크로드 GPU 수요(3장)보다 플랫폼 최소요건(GPU 호스트 3대, 컨트롤 플레인 3, HA)이 노드 수를 지배**합니다. 소규모 AI 서비스일수록 "최소 구성 비용"이 단가의 큰 부분이 됩니다.

> HA 단계화: PoC·파일럿은 단일 vSphere Zone·3 호스트로 시작하고, 프로덕션 HA가 필요하면 3-Zone(Zone당 최소 3 호스트)으로 확장합니다([04.1](04-vks-cluster-sizing.md#41-vks-클러스터-토폴로지-개요)). GPU 노드 오토스케일은 부팅·드라이버 기동 지연이 크므로 min에 헤드룸을 둡니다([04.4](04-vks-cluster-sizing.md#44-오토스케일-헤드룸버스트스케일-지연)).

GPU Operator 데몬셋 오버헤드(노드당 상시 vCPU·RAM)는 노드 가용 자원에서 차감해야 합니다([04.3](04-vks-cluster-sizing.md#43-노드-사양-산정-노드-크기-vs-노드-수)).

| 갈래 | 논리 용량(어림) | 근거 |

| NIM(NVIDIA Inference Microservices) 컨테이너 이미지 | 약 50GB(확인 필요) | 05.1 |
| 문서 코퍼스 | 100GB × 1.5 ≈ 150GB | 05.1 |
| 벡터 인덱스(pgvector) | 원시 30GB + HNSW 60GB ≈ 90GB | [05.2](05-storage-network-sizing.md#52-벡터-인덱스-용량--dsm-pgvector-산정-②-연계) |
| 로그·관측(90일) | 약 30GB | 05.1 |
| **논리 합계** | **약 368GB** | 합산 |
| **필요 가용 용량** | × 1.5(보호 오버헤드) ≈ **약 0.6TB** | [05.3](05-storage-network-sizing.md#53-스토리지-성능--모델-로딩과-iops) |

## 8.6 단계 6 — TCO 골격 (본문 07 적용)

[07](07-tco-cost-model.md)의 분해에 사이징 결과를 입력합니다. **단가는 비우고**(견적), 산정 입력만 채웁니다.

| 대분류 | 사이징 입력(이 시나리오) | 단가 |
|---|---|---|
| VCF 코어 구독 | GPU 호스트 3 + 일반 3 + 컨트롤플레인의 물리 코어 합 × **코어 최소수량 규칙** | (견적) |
| NVAIE(NVIDIA AI Enterprise, GPU당 과금) | 설치된 물리 GPU 전수(약 3) | (NVIDIA 견적) |
| DSM(Data Services Manager) | VCF 포함 권한(벡터DB용 pgvector) | (조건 확인) |
| GPU 하드웨어 | 약 3 GPU ÷ 상각연수 | (견적) |
| 서버·스토리지·네트워크 | 노드 ~6 호스트 + vSAN 약 0.6TB | (견적) |
| OpEx | 운영 FTE, 전력(GPU 소비전력 × PUE), 상면 | (사이트 데이터) |

산정 함정 점검([07.2](07-tco-cost-model.md#72-소프트웨어-라이선스구독-비용)):

- PAIF는 VCF 구독에 포함 → 별도 제품으로 **이중 계상 금지**.
- NVAIE는 vGPU 분할과 무관하게 **물리 GPU 수 기준**.
- VCF 코어는 사이징 물리 코어를 그대로 쓰지 말고 **코어 최소수량 규칙** 반영.

> 빈 단가 칸을 채우는 방법(무엇을 견적 요청할지)은 부록 A3(RFQ 체크리스트, Phase 3 예정)에서 다룹니다. 퍼블릭 클라우드와의 손익분기 개념은 [07.6](07-tco-cost-model.md#76-퍼블릭-gpu-클라우드-vs-온프레미스-paif-비교-프레임)을 참조하세요.

## 8.7 결과 요약과 다음 행동

| 산출물 | 이 시나리오 결과 | 확정 경로 |
|---|---|---|
| 설계 피크 동시성 | 약 21 | 파일럿 실측([06.5](06-capacity-planning.md#65-poc--파일럿--프로덕션-용량-로드맵)) |
| 모델 | 8B급 FP16(양자화 옵션) | PoC 품질 평가 |
| 총 GPU | 약 3장 | PoC 처리량 실측 → A1 가정치 교체 |
| 클러스터 | 9 노드(컨트롤 3 + 일반 3 + GPU 3) | 04.7 실측 |
| 스토리지 | 가용 약 0.6TB | 05.6 실측 |
| TCO | 코어/GPU/용량 입력 확정, 단가는 견적 | 07.8 실측·견적 |

### 이 예제가 보여주는 것

1. **방법은 결정론적이다.** 입력(A2) → 자원(02·03) → 노드(04) → 스토리지(05) → 비용(07)이 끊김 없이 이어집니다.
2. **RAG의 GPU 수요는 작을 수 있다.** 지식이 벡터DB에 있어 모델·GPU를 무작정 키울 필요가 없습니다.
3. **소규모일수록 플랫폼 최소요건이 비용을 지배한다.** GPU 3장짜리 서비스도 PAIF·HA 최소 구성이 노드·라이선스 코어를 끌어올립니다.

> 모든 숫자는 부록 A1의 1차 가정치를 따른 값입니다. 다음 행동은 **PoC**입니다: 최소 GPU로 8B 후보의 품질·VRAM·처리량을 실측해 위 표의 가정을 교체하세요([06.5](06-capacity-planning.md#65-poc--파일럿--프로덕션-용량-로드맵)). 자사 입력으로 다시 풀려면 [부록 A2.6 워크시트](../appendix/A2-inputs-and-defaults.md#a26-입력값-워크시트-복사해서-채우기)를 채우고 8.1부터 반복하세요.
