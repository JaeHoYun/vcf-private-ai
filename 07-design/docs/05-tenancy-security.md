# 05 — 설계 결정: 멀티테넌시·보안 설계

[← 목차로](../README.md)

이 문서는 **설계 시점에 갈리는** 테넌시·신원·격리 결정만 다룹니다. 위협 모델, 접근 통제 상세, 공급망·데이터 거버넌스·감사 등 보안 운영의 깊이는 ⑤ 보안·거버넌스 가이드로 위임합니다. 세 가지 설계 결정을 다룹니다.

> 본 문서의 수치·동작은 작성 시점(2026-06) VCF 9.1 / PAIF 9.1 / PAIS 2.1 기준이며, 적용 전 공식 문서로 재확인하시기 바랍니다.

## 5.1 결정: 멀티테넌시 격리 — soft(네임스페이스) vs hard(클러스터·도메인 분리)

**무엇을 정하나** — 여러 팀·테넌트가 한 플랫폼을 공유할 때 격리를 어디까지 강제할지.

- **경로 A — soft 격리**: vSphere Supervisor 네임스페이스 / 쿠버네티스 네임스페이스 + RBAC + 리소스 쿼터. 자원 효율이 높지만 커널·노드를 공유하므로 격리는 논리 수준.
- **경로 B — hard 격리**: 테넌트별 VKS 클러스터 또는 별도 워크로드 도메인. 강한 격리(자원·장애·보안 경계)이나 자원·운영 오버헤드가 큼.
- **보강 — 네트워크 격리**: 어느 경로든 NSX VPC + 마이크로세그멘테이션으로 테넌트 간 트래픽을 분리(→ [4.1](04-network-storage-availability.md)).

| 차원 | soft(네임스페이스) | hard(클러스터·도메인) |
|------|-------------------|----------------------|
| 격리 강도 | 논리(공유 커널) | 물리·강함 |
| 자원 효율 | 높음 | 낮음(중복) |
| 운영 오버헤드 | 낮음 | 높음 |
| 폭발 반경 | 큼 | 작음 |
| 적합 | 신뢰 내부 팀 | 규제·외부·고민감 테넌트 |

**상황별 권고**
- 신뢰 가능한 내부 팀 다수, 비용 효율 우선 → **soft + VPC 마이크로세그**
- 규제·외부 테넌트·강한 격리 의무 → **hard(테넌트별 클러스터/도메인)**

### 5.1.1 전사 확장 시 멀티테넌트 기반 설계

위의 soft/hard 결정 틀은 그대로 둡니다. 다만 한두 팀이 아니라 전사 수십 개 팀·사업부로 테넌트가 늘어나면, 같은 결정을 **테넌트별로** 반복하지 않고 **플랫폼 차원의 기반**으로 묶어야 합니다. 전사 확장은 [02 대(Scale/Regulated) 블루프린트](02-reference-blueprints.md#24-대scale--regulated)를 출발점으로 삼고, 아래 다섯 가지를 함께 설계합니다.

**테넌시 경계의 계층** — VCF Automation의 Project가 주 테넌시 경계(primary tenancy boundary)이며, 프로비저닝 시 Supervisor 클러스터에 대응하는 vSphere 네임스페이스가 생성되어 거버넌스 정책을 실제 워크로드 자원(쿼터)으로 강제하는 경계로 동작합니다. All Apps 조직(Organization)을 쓰는 경우 조직 간 자원을 완전 격리하고, 그 아래 Project·vSphere 네임스페이스 수준에서 추가로 격리합니다. 즉 전사 확장에서는 soft/hard가 단일 선택이 아니라 **조직 → Project → 네임스페이스**의 다단 경계로 조합됩니다.

**언제 soft에서 hard로 넘어가나 (격리 선택의 임계)** — 테넌트 수 자체보다 **신뢰 경계와 규제 요건**이 임계를 가릅니다. 다음 중 하나라도 해당하면 그 테넌트만 hard로 끌어올리는 혼합(hybrid) 구성을 권고합니다.

| 신호 | soft 유지 | hard로 승격 |
|------|----------|------------|
| 신뢰 관계 | 내부 동일 보안 등급 | 외부·계열 분리·상호 비신뢰 |
| 규제·데이터 주권 | 일반 | 방산·금융·개인정보 분리 의무 |
| 자원 경합·소음 이웃 | 쿼터로 흡수 가능 | 성능 보장(QoS) 필수 |
| 폭발 반경 허용 | 팀 단위 | 사업부·법인 단위 격리 필요 |

전사 전체를 hard로 가져가면 자원·운영 오버헤드가 급증하므로, **기본은 soft 다수 + 고민감 테넌트만 hard**가 비용·격리의 균형점입니다.

**자원 배분 — 쿼터·네트워크·GPU**
- **네임스페이스 쿼터**: VI 관리자가 네임스페이스별 CPU(GHz)·RAM(GB)·스토리지(TB, VM Storage Policy 경유)를 명시적으로 한정합니다(예: 한 사업부 네임스페이스를 512GB RAM·2TB vSAN으로 제한). 전사 확장에서는 이 쿼터가 사업부별 예산·우선순위를 강제하는 1차 통제점입니다.
- **NSX VPC**: 네트워크·보안 정책 격리는 NSX Project·VPC가 담당합니다. 테넌트마다 VPC를 두어 주소 공간·정책을 분리하고, hard 테넌트는 VPC + 마이크로세그멘테이션으로 동서 트래픽까지 차단합니다(→ [4.1](04-network-storage-availability.md)). VKS 3.6의 선언형 멀티 NIC는 primary(eth0, 불변 백본)와 secondary(eth1–eth9, 노드풀별 추가)를 구분하며, NetworkPolicy(L4)만으로는 노드 NIC 수준 분리나 클러스터 외부 트래픽 통제가 어려우므로 토폴로지 격리를 secondary NIC로 보완합니다.
- **GPU 배분(MIG)**: 멀티테넌트에서는 MIG(Multi-Instance GPU)를 기본으로 합니다. 물리 GPU를 분할해 frame buffer·SM·L2 캐시·메모리 컨트롤러를 하드웨어 수준으로 격리하므로, 한 테넌트의 부하가 다른 테넌트에 영향을 주지 않는 엄격한 QoS(strict QoS)를 보장합니다(예: A100 80GB에서 1g.10gb는 1/7 컴퓨트 + 10GB). 반대로 타임슬라이싱은 활용률은 좋지만 성능이 동거 테넌트 활동에 따라 흔들려 QoS 보장이 없으므로 신뢰 내부 팀 간에만 권고합니다. 동일 물리 GPU에 MIG와 타임슬라이스 프로필은 공존할 수 없습니다(→ [3.2 GPU 공유](03-compute-gpu-topology.md)).

**테넌트 온보딩/오프보딩** — 온보딩은 Cloud Admin이 VCF Automation에서 Project를 생성하면, 개발자가 SSO 인증 후 할당된 Project만 조회하고, 인프라가 대응 vSphere 네임스페이스를 쿼터와 함께 자동 프로비저닝하는 흐름입니다(신원·SSO는 [5.2](#52-결정-identity--내장-vs-외부-idp-페더레이션)). 전사 확장에서는 이 절차를 표준 템플릿(쿼터 등급·VPC·GPU 프로필·기본 권한)으로 묶어 반복 가능하게 만들고, 오프보딩 시에는 자원 회수·데이터 폐기·권한 말소를 같은 템플릿의 역순으로 정의합니다. 모델 접근은 Model Gallery(Harbor 기반)의 Project 수준 접근 통제로 학습·튜닝 데이터 권한을 분리합니다.

**showback(사용량 가시화)** — Private AI Model·GPU Metrics가 GPU 활용률·메모리 압박·모델 수준 가시성을, AI 관측·거버넌스가 time-to-first-token·토큰 처리량 등을 전체 인프라와 동일 콘솔에서 노출합니다. 이 메트릭을 테넌트(Project·네임스페이스) 단위로 집계하면 사업부별 사용량을 보여주는 showback의 토대가 됩니다. 다만 작성 시점(2026-06) 별도 차지백 과금 메커니즘은 공식 확인되지 않았고, VCF Automation 멀티테넌시 모드에서 vSphere Pod의 임시(ephemeral) 스토리지 소비가 네임스페이스 스토리지 쿼터에 집계되지 않는 알려진 제약이 있어, 차지백 정산보다는 **메트릭 기반 showback(가시화·내부 배분)** 수준으로 설계하기를 권고합니다.

> GPU 분할 모드(MIG·타임슬라이싱·하이브리드)의 정식 명칭·지원 범위, 임시 스토리지 쿼터 집계, showback/차지백 기능 여부는 릴리스에 따라 다르므로 적용 전 공식 문서로 재확인하시기 바랍니다.

## 5.2 결정: Identity — 내장 vs 외부 IdP 페더레이션

**무엇을 정하나** — 플랫폼 인증을 VCF 내장으로 둘지, 기업 표준 IdP에 페더레이션할지.

- **경로 A — 내장(embedded Identity Broker)**: vCenter에 내장된 VCF Identity Broker로 로컬·간단하게 시작. 소규모·독립 환경에 빠름.
- **경로 B — 외부 IdP 페더레이션**: VCF Management Services의 외부 Identity Broker + 기업 IdP(Entra ID·Okta·Ping 등)와 OIDC/OAuth 2.0 페더레이션. 중앙 집중 접근 통제와 SSO를 제공. VCF 9.1은 내장 vs 외부 Identity Broker를 선택할 수 있음.

| 차원 | 내장 | 외부 IdP 페더레이션 |
|------|------|---------------------|
| 도입 속도 | 빠름 | 중(연동 작업) |
| 중앙 접근 통제·SSO | 제한 | 강함 |
| 기업 표준 부합 | 낮음 | 높음(OIDC) |
| 운영·감사 | 분산 | 일원화 |

**상황별 권고**
- 소규모·독립·빠른 시작 → **내장 Identity Broker**
- 기업 표준 IdP·SSO·중앙 접근 통제·규제 → **외부 IdP 페더레이션**

## 5.3 결정: 에어갭 수준 — 완전 격리 vs 프록시 경유 vs 온라인

**무엇을 정하나** — 플랫폼이 외부 네트워크(모델·드라이버·이미지 출처)에 어디까지 연결될지.

- **경로 A — 완전 에어갭**: 인터넷 단절. 내부 미러(Harbor)와 Artifact Mirroring Tool(아티팩트 미러링 도구)로 모델뿐 아니라 NIM·드라이버·컨테이너 이미지·Helm 차트 등 아티팩트 전반을 미러링해 수동 반입. 최고 보안, 운영 부담 최대.
- **경로 B — 프록시 경유**: 제한적 아웃바운드를 프록시·허용목록으로 통제해 NGC·모델 레지스트리에서 pull. 보안과 운영의 균형.
- **경로 C — 온라인**: 직접 인터넷 연결. 운영 부담 최소, 보안 노출 최대.

| 차원 | 완전 에어갭 | 프록시 경유 | 온라인 |
|------|------------|------------|--------|
| 보안·격리 | 최고 | 중·높음 | 낮음 |
| 운영 부담 | 최대(수동 반입) | 중 | 최소 |
| 모델 갱신 속도 | 느림 | 보통 | 빠름 |
| 규제 적합 | 방산·국가급 | 일반 기업 | 낮은 민감도 |

**상황별 권고**
- 방산·금융·국가 규제, 완전 격리 의무 → **완전 에어갭(미러 + Artifact Mirroring Tool)**
- 일반 기업 보안 기준 → **프록시 경유**
- 내부 PoC·낮은 민감도 → **온라인**

> CLI는 두 갈래입니다 — 에어갭 미러링은 `vcf pais amt pull/push`(VCF CLI 명령 레퍼런스에는 누락, Disconnected Environment 배포 문서에 명시), 모델 적재는 `vcf pais models`(push/pull/list/delete). 세부 플래그·동작은 릴리스에 따라 달라질 수 있으므로 적용 전 공식 문서와 ① Artifact Mirroring Tool 런북으로 재확인하시기 바랍니다(작성 시점 기준).
