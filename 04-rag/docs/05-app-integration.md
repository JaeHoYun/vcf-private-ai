# 05 — 앱 통합 패턴

[← 목차로](../README.md)

추론을 실제 사내 서비스로 감싸는 단계입니다. ③의 4-Tier 앱 구조를 RAG 시나리오에 맞춰 구체화합니다.

## 5.1 4-Tier 구조

```
[클라이언트]  웹/모바일/사내 메신저 봇
     │
[BFF/게이트웨이]  인증·세션·요청정형·요청 속도 제한
     │
[RAG 오케스트레이션]  03 검색 + 04 추론  (경로 B면 여기서 조립 / 경로 A면 Agent 호출)
     │
[PAIS / pgvector]  임베딩·완성·벡터검색  (②③)
```

핵심은 **RAG 오케스트레이션 계층을 독립**시키는 것입니다. 검색 전략·프롬프트·리랭킹을 이 계층에 모아두면, 모델·인덱스를 바꿔도 클라이언트는 그대로입니다.

## 5.2 base_url 스위치 — 점진 전환

PAIS가 OpenAI 호환이라, 이미 외부 LLM을 쓰던 앱은 **`base_url`과 키만 사내 PAIS로 교체**하면 데이터가 사내에 머무는 추론으로 전환됩니다(③). RAG 도입 시 권장 순서:

1. 기존 LLM 호출을 PAIS 완성 API로 스위치(데이터 사내화 먼저 확보)
2. 임베딩+pgvector 인덱싱 구축(02)
3. 오케스트레이션 계층에 검색→조립 삽입(03)
4. 평가로 품질 확인 후 전사 확대(06)

## 5.3 인증·인가

- **서비스 인증**: 앱→PAIS 호출은 토큰 기반(③ 인증 문서). 토큰은 시크릿 매니저로 관리, 코드/리포에 하드코딩 금지.
- **사용자 인가 → 검색 필터**: 최종 사용자의 권한(부서·등급)을 03의 메타데이터 필터로 전달해, **사용자가 볼 수 있는 문서만 검색**되게 합니다. 인가를 추론이 아니라 **검색 단계**에서 거는 것이 핵심 — 권한 밖 문서는 프롬프트에 애초에 안 들어갑니다.

## 5.4 멀티턴 세션

문서 Q&A는 후속 질문("그럼 그건?")이 흔합니다.

- **세션 상태**: 대화 이력을 BFF(Backend For Frontend, 프론트엔드 전용 백엔드)/오케스트레이션 계층에 보관. 경로 A(Agent)는 세션을 캡슐화할 수 있어 활용(④4.1).
- **후속 질문 재작성(query rewriting)**: 대명사·생략이 많은 후속 질문은 그대로 임베딩하면 검색이 빗나갑니다. 직전 맥락을 합쳐 **독립형 질문으로 재작성**한 뒤 검색하면 정밀도가 크게 오릅니다.
- **컨텍스트 예산**: 이력 + 근거 + 질문이 윈도우를 넘지 않게 이력 요약/절단.

## 5.5 사내 메신저 봇 패턴 (레퍼런스 시나리오)

가장 흔한 배포 형태입니다.

- 메신저 → BFF(웹훅) → 오케스트레이션 → PAIS, 응답을 스트리밍 또는 분할 전송.
- 출처는 메시지 하단에 링크 카드로. 권한은 메신저 사용자 ID ↔ 사내 디렉터리 매핑으로 검색 필터에 연결.
- 피드백 버튼(좋아요/싫어요)을 달아 06의 평가 데이터로 수집.

## 5.6 핵심 결정 요약

| 결정 | 기본 권장 |
|------|-----------|
| 오케스트레이션 분리 | 독립 계층으로(모델·인덱스 교체 격리) |
| 전환 순서 | base_url 스위치 → 인덱싱 → 검색 삽입 |
| 인가 위치 | 추론이 아니라 검색 필터에서 |
| 후속 질문 | 독립형 질문 재작성 후 검색 |
| 피드백 | 좋아요/싫어요 수집 → 평가 루프 |
