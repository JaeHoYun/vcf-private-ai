# 03 — 검색·컨텍스트 조립

[← 목차로](../README.md)

질문이 들어온 직후, 답변 생성 전의 단계입니다. 질문을 받아 **모델에게 넘길 근거 묶음**을 만드는 단계로, 답변 품질과 환각률을 좌우합니다.

## 3.1 질문 임베딩 & 유사도 검색

질문을 **인덱싱 때와 같은 임베딩 모델**로 벡터화한 뒤 pgvector에서 가장 가까운 청크 K개를 가져옵니다.

```sql
-- 개념 예시 (연산자·튜닝은 ② 참조)
SELECT content, metadata, embedding <=> :query_vec AS distance
FROM kb_chunks
WHERE metadata->>'access' = ANY(:user_scopes)   -- 접근 통제 필터
ORDER BY embedding <=> :query_vec
LIMIT :k;
```

- **K(top-k)**: 너무 작으면 근거 누락, 너무 크면 노이즈·토큰 낭비. 보통 **5–10**에서 시작.
- **거리 연산자**: 임베딩 모델이 권장하는 것(코사인 등)을 인덱스와 일치시켜 사용.
- **메타데이터 필터 선적용**: 접근등급·부서·기간 필터를 검색에 결합해, 권한 밖 문서가 애초에 후보에 안 들게 합니다(보안 + 정밀도).

## 3.2 순수 벡터 검색의 한계와 하이브리드

벡터 검색은 의미는 잘 잡지만 **정확한 키워드·고유명사·코드·약어**에 약합니다(예: 특정 모델 번호, 사번 규칙). 이때 **하이브리드 검색**을 씁니다.

- **키워드(BM25 같은 전문검색 알고리즘) + 벡터** 결과를 결합. PostgreSQL의 전문검색(`tsvector`)과 pgvector를 함께 사용할 수 있습니다(②).
- 결합 방식: 가중합 또는 **RRF(Reciprocal Rank Fusion)** 로 두 랭킹을 병합.

> PAIS **Agent Builder(경로 A)** 를 쓰면 검색·조립의 상당 부분이 플랫폼에서 처리됩니다. 하이브리드·리랭킹의 네이티브 지원 범위는 PAIS 2.1 공식 문서로 확인하고, 부족하면 **앱 계층(경로 B)** 에서 보완하는 것으로 설계하세요.

## 3.3 리랭킹

1차 검색(속도는 빠르지만 정밀도는 낮은 방식)으로 후보 N개(예: 50)를 넓게 뽑고, **리랭커 모델**로 질문–청크 적합도를 정밀 재정렬해 상위 K개(예: 5)만 남깁니다. NVIDIA 기반 PAIF는 임베딩·**리랭킹**·추론에 GPU를 활용합니다.

```
1차 검색 top-50 (벡터/하이브리드) → 리랭커 → top-5 → 컨텍스트 조립
```

- 효과: 검색 재현율(recall)은 유지하면서 최종 정밀도(precision)를 끌어올려 환각·노이즈를 줄입니다.
- 비용: 리랭커 추론이 추가되므로 지연·GPU 사용↑. 후보 N과 지연을 06에서 균형 맞춥니다.
- 가용성: 리랭커 모델은 ③ Model Runtime에서 서빙하거나 Agent Builder가 제공하는 범위를 사용. 구체 지원은 공식 문서 확인.

## 3.4 컨텍스트 조립

선별된 청크를 프롬프트에 넣을 형태로 묶습니다.

- **출처 메타데이터 동봉**: 각 청크에 `[출처: 문서명/경로]` 식 라벨을 붙여 넣으면, 모델이 답변에 인용을 달기 쉽고 04의 출처 표기로 이어집니다.
- **컨텍스트 윈도우 예산**: (시스템 프롬프트 + 근거 청크 + 질문 + 답변 여유)가 모델 컨텍스트 한도 안에 들어야 합니다. 청크가 길거나 K가 크면 초과합니다 → K·청크 크기 조정 또는 요약 압축.
- **순서**: 일부 모델은 컨텍스트 중간 정보를 놓치는 경향("lost in the middle")이 있어, 가장 적합한 근거를 앞/뒤에 배치하는 것이 도움이 될 수 있습니다(모델별 06에서 검증).
- **근거 없음 처리**: 검색 점수가 임계 이하면 "관련 문서를 찾지 못했습니다"로 폴백하도록 설계 — 억지 생성(환각)을 막는 가장 효과적인 장치입니다.

## 3.5 프롬프트 골격 (예시)

```text
[시스템] 너는 사내 문서 기반 도우미다. 아래 '근거'에 있는 내용만으로 답하고,
근거가 부족하면 모른다고 답하라. 답변 끝에 사용한 출처를 나열하라.

[근거]
(1) [출처: 보안정책-v3 / 4.2절] <청크 내용>
(2) [출처: 온보딩가이드 / 비밀번호] <청크 내용>
...

[질문] {사용자 질문}
```

## 3.6 보안 — 프롬프트 인젝션 방어와 입력 살균

RAG에는 일반 챗봇에 없는 고유한 공격면이 있습니다. **검색된 청크 자체가 인젝션 벡터**라는 점입니다. RAG는 외부(사내 문서)에서 가져온 텍스트를 프롬프트에 끼워 넣어 모델에 먹이는데, 그 문서 안에 "이전 지시를 무시하고 …하라" 같은 **숨겨진 명령**이 들어 있으면 모델이 그것을 사용자 지시로 오인할 수 있습니다. 이것이 OWASP가 LLM01로 꼽은 **간접 프롬프트 인젝션(Indirect Prompt Injection)** 이며, OWASP는 "공격자가 RAG가 참조하는 저장소의 문서를 수정해 두면, 사용자 질의가 그 오염된 콘텐츠를 검색했을 때 악성 지시가 LLM 출력을 변조한다"고 명시합니다([OWASP LLM01:2025 Prompt Injection](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)). 신뢰하던 데이터 소스 자체가 공격 경로가 됩니다. 벡터 저장소·임베딩 파이프라인에 대한 오염·미흡한 격리·미살균 검색결과 주입은 별도 항목인 [OWASP LLM08:2025 Vector and Embedding Weaknesses](https://genai.owasp.org/llmrisk/llm082025-vector-and-embedding-weaknesses/)로도 다뤄집니다.

방어는 한 겹으로 끝나지 않습니다. OWASP가 권고하는 **다층 방어(defense-in-depth)** 를 RAG 단계에 매핑하면 다음과 같습니다.

- **시스템 프롬프트와 데이터의 분리·역할 고정**: 시스템 메시지로 모델의 역할·권한·한계를 못 박고, 검색된 근거는 어디까지나 **참고 데이터일 뿐 지시가 아님**을 명시합니다. 검색 청크는 사용자/시스템 역할과 섞지 말고 별도의 `[근거]` 블록(3.5)에 구획해 넣어, 데이터 영역의 텍스트를 명령으로 해석하지 않도록 경계를 그립니다.
- **검색결과·입력 살균(sanitization)**: 인덱싱 시점과 검색 직후 모두에서, 청크에 섞인 프롬프트형 문자열("위 지시를 무시", "system:", 역할 토큰 위장 등)·숨김 텍스트·제어문자·마크업을 정규화/제거합니다. 사용자 입력도 동일하게 살균합니다(직접 인젝션 대비). 의미 필터와 문자열 검사를 함께 적용하라는 것이 OWASP의 권고입니다.
- **최소 권한 + 검색 단계 인가**: 모델·에이전트에 부여하는 도구/권한을 최소화하고, 문서 접근 통제는 **검색 쿼리에 권한 필터를 선결합**(3.1)해 권한 밖 문서가 애초에 컨텍스트에 들어오지 않게 합니다. OWASP LLM08은 벡터·임베딩 스토어에 **권한 인식(permission-aware) 접근 통제**와 데이터셋의 논리적 분할을 적용하라고 권고합니다.
- **지식 베이스 무결성 검증**: 인덱싱 소스(지식 베이스, KB)에 대한 데이터 검증 파이프라인을 두고, 숨겨진 명령·오염(poisoning) 여부를 주기적으로 감사합니다(LLM08). 소수의 조작 문서만으로도 응답을 변조할 수 있다는 연구가 보고되어 있어, 출처 신뢰도 관리가 중요합니다.
- **고위험 동작에는 휴먼인더루프 + 적대적 테스트**: 검색·답변이 외부 행위(메일 발송, 명령 실행 등 도구 호출)로 이어지는 경우 사람 승인을 끼우고, 정기적으로 인젝션 시나리오를 넣은 적대적 테스트(06의 회귀셋에 포함)를 돌립니다.

> **출력단 가드레일**(PII·민감정보 마스킹, 유해/시스템 프롬프트 누출 필터)은 답변 생성 직후에 거는 별도 방어선으로, [04 추론 통합 4.6](04-inference-integration.md#46-출력-가드레일--민감정보출력-안전)에서 다룹니다. 입력측(여기 3.6)과 출력측(04.6)을 모두 두는 것이 다층 방어의 핵심입니다.
>
> **PAIS 연계**: PAIS 2.1이 제공하는 네이티브 가드/필터 기능의 범위(인젝션 탐지·출력 필터 내장 여부)는 릴리스별로 다를 수 있으므로 [PAIS 공식 문서](https://developer.broadcom.com/xapis/vmware-private-ai-service-api/latest/)로 **확인 필요**합니다. 플랫폼이 일부를 제공하더라도, 위 방어는 앱 오케스트레이션 계층에서 독립적으로 한 번 더 적용하는 것을 권장합니다.

## 3.7 핵심 결정 요약

| 결정 | 기본 권장 |
|------|-----------|
| top-k | 5–10에서 시작, 06으로 조정 |
| 하이브리드 | 키워드 의존 질의 많으면 도입(RRF) |
| 리랭킹 | 품질 병목 시 1차 50 → 리랭크 → 5 |
| 근거 없음 | 임계 미달 시 폴백(억지 생성 금지) |
| 접근 통제 | 검색 쿼리에 권한 필터 선결합 |
| 인젝션 방어 | 시스템/데이터 분리 + 청크·입력 살균 + KB 무결성 감사(3.6) |
