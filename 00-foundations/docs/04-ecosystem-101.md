# 04 — 생태계 SW 101: GPU 위에서 도는 AI 소프트웨어 스택

이 문서는 인프라 엔지니어가 "GPU만 꽂으면 모델이 도는가"라는 흔한 오해를 풀고, 물리 GPU(03) 위에 어떤 소프트웨어가 겹겹이 쌓여야 모델이 비로소 동작하는지를 잡는 입문서입니다. 각 계층의 이름과 역할, 그리고 그것들을 묶어 파는 생태계(NVIDIA AI Enterprise·모델 산출물·레지스트리)를 어휘 수준에서 정리합니다. 서빙 엔진 내부 동작은 시리즈 ③, 버전 호환성 매트릭스는 ①, 모델 공급망 통제는 ⑤가 다룹니다.

비유는 직관을 잡기 위한 것이며, 정확한 정의·버전·절차는 본편과 각 벤더의 공식 문서를 따릅니다.

## 4.1 GPU는 소프트웨어 스택이 받쳐 줘야 동작한다

물리 GPU를 서버에 꽂고 하이퍼바이저·노드까지 노출했다고(03) 모델이 바로 도는 것은 아닙니다. 그 위에 드라이버부터 추론 엔진까지 여러 겹의 소프트웨어가 **순서대로, 서로 버전이 맞물려** 올라가야 합니다.

멘탈모델로 보면 스토리지와 같습니다. 디스크를 꽂는다고 끝이 아니라 드라이버·파일시스템·볼륨 매니저가 받쳐 줘야 쓸 수 있듯, GPU도 그 위의 소프트웨어 스택이 갖춰져야 연산 자원이 됩니다.

## 4.2 GPU 소프트웨어 스택 — 다섯 계층

아래에서 위로, 각 계층은 바로 아래 계층에 기대어 동작합니다.

```
모델 / 애플리케이션
   ▲
[5] 추론 엔진      vLLM · llama.cpp · Infinity   ← 모델을 실제로 실행·서빙
[4] 프레임워크     PyTorch 등                     ← 모델을 표현하고 연산으로 변환
[3] 가속 라이브러리 cuDNN · NCCL                   ← 딥러닝 연산·멀티 GPU 통신
[2] CUDA          GPU 연산 플랫폼                  ← GPU에 계산을 시키는 공통 토대
[1] GPU 드라이버   커널이 GPU를 인식               ← 가장 아래, 하드웨어 접점
   ▲
물리 GPU (03)
```

- **[1] GPU 드라이버** — 운영체제(노드)가 GPU 하드웨어를 인식하게 하는 가장 아래 계층입니다. 스토리지 드라이버와 같은 자리입니다.
- **[2] CUDA** — GPU에 일반 연산을 시키기 위한 NVIDIA의 플랫폼입니다. 멘탈모델로는 "GPU를 프로그래밍하는 공통 언어이자 런타임"으로, 위 계층 거의 전부가 이 위에서 돌아갑니다.
- **[3] 가속 라이브러리** — CUDA 위에서 자주 쓰는 연산을 최적화해 둔 묶음입니다. **cuDNN**은 딥러닝 연산을, **NCCL**은 여러 GPU 간 고속 통신을 담당합니다(§3.6의 멀티 GPU가 실제로 기대는 라이브러리입니다).
- **[4] 프레임워크** — **PyTorch** 같은 도구로, 모델을 표현하고 그것을 GPU 연산으로 바꿔 줍니다. 개발자가 직접 마주하는 계층입니다.
- **[5] 추론 엔진** — 완성된 모델을 받아 실제로 실행하고 API로 내보내는 계층입니다. 시리즈에서는 **vLLM**(GPU·운영 추론), **llama.cpp**(CPU 추론), **Infinity**(임베딩 생성)를 씁니다. 엔진의 내부 동작(연속 배칭·KV 캐시 등)은 ③이 다룹니다.

**인프라 관점의 핵심은 이 계층들의 버전이 서로 맞물린다는 점입니다.** 드라이버·CUDA·라이브러리·엔진 버전이 어긋나면 위가 멀쩡해도 모델이 안 뜹니다. 이 "버전 인터락"을 매트릭스로 관리하는 일이 ①(GPU enablement)의 핵심 작업이며, 이 입문서는 "이런 계층이 맞물려 있다"는 지도까지만 그립니다.

## 4.3 NVIDIA AI Enterprise(NVAIE) — 스택을 묶어 파는 구독

위 스택의 상당 부분(드라이버·라이브러리·일부 런타임)은 개별로도 존재하지만, **NVIDIA AI Enterprise(NVAIE)** 는 이를 검증·인증해 묶고 상용 지원을 붙인 소프트웨어 구독입니다.

인프라·구매 관점에서 NVAIE가 중요한 이유는 두 가지입니다.

- **지원 경계와 비용** — 어디까지가 벤더 지원 범위인지, 라이선스 비용이 어떻게 붙는지를 가르는 항목입니다. GPU 도입 견적에서 자주 등장합니다.
- **할당 모드와의 관계** — GPU 분할·공유 방식(§3.7의 vGPU 등)에 따라 NVAIE 라이선스가 필요한 지점이 갈립니다. 다만 9.1의 Enhanced DirectPath I/O로 **전용(passthrough) GPU 액세스에는 NVAIE 없이** vMotion까지 유지하는 경로가 생겼습니다. 정확한 라이선스 분기는 ①이 다룹니다.

## 4.4 모델도 배포해야 할 소프트웨어 자산이다

모델은 추상적인 무엇이 아니라 **파일로 존재하는 배포 아티팩트**입니다. 컨테이너 이미지(§2.1.2)를 다루듯 버전·저장·반입을 관리해야 합니다.

- **오픈웨이트(open-weight)** — 가중치가 공개되어 받아서 사내 GPU에서 직접 실행할 수 있는 모델(Llama·Mistral·Qwen 등)입니다. 가중치를 내려받아 경계 안에서 돌릴 수 있다는 점이 Private AI의 전제입니다.
- **모델 포맷** — 같은 모델도 저장 형식이 다릅니다. **safetensors**(가중치 표준 저장 형식), **GGUF**(llama.cpp용으로 양자화된 단일 파일), **NIM**(NVIDIA 추론 컨테이너 런타임 형식)이 대표적입니다.
- **외부 출처** — 오픈웨이트 모델은 흔히 **Hugging Face** 같은 외부 허브에서 배포됩니다. 사내로 들일 때는 이 외부 산출물을 어떻게 반입·검증할지가 인프라·보안 문제로 바뀝니다.

## 4.5 모델을 어디서 받아 어디에 두나 — 레지스트리와 폐쇄망

컨테이너 이미지에 사내 레지스트리가 필요하듯, 모델·이미지 같은 AI 아티팩트도 사내에 두고 관리할 저장소가 필요합니다.

- **레지스트리(Harbor)** — VCF 환경에서 컨테이너 이미지·아티팩트를 보관하는 사내 레지스트리입니다. 외부에서 받은 모델·이미지를 사내에 들여 두는 저장소 역할을 합니다.
- **폐쇄망(air-gapped) 반입** — 외부 인터넷이 차단된 환경에서는 외부 허브에 직접 닿을 수 없습니다. PAIS 2.1의 **Artifact Mirroring Tool** 은 이런 폐쇄망에 GPU 기반 Model Endpoint와 에이전트를 포함한 완전한 Private AI 기능을 들여 구동하게 해 주는 도구입니다. 방산·금융·공공처럼 외부 반출이 불가한 환경의 핵심입니다.

멘탈모델은 "사내 패키지 미러·아티팩트 저장소"입니다. 외부에서 가져온 것을 사내에 한 번 받아 두고, 그 안에서만 배포하는 구조입니다. 모델 공급망의 무결성·서명·취약점 검증은 시리즈 ⑤(보안)가 다룹니다.

## 4.6 PAIS가 이 생태계를 묶는 방식

앞 절들의 조각(스택·모델 산출물·레지스트리)을 매번 손으로 조립하지 않도록, PAIS는 이를 관리형 흐름으로 묶습니다.

- **Model Gallery** — 외부에서 받은 모델을 사내로 반입해 두는 곳입니다(어느 포맷으로 둘지 여기서 결정).
- **Model Runtime** — 반입된 모델을 추론 엔진(vLLM·llama.cpp·Infinity)으로 실행하는 계층입니다.
- **Model Endpoint** — 실행된 모델을 **OpenAI 호환 API** 로 내보내는 운영 접점입니다. 엔진이 무엇이든 앱은 같은 방식으로 호출합니다.

즉 "외부 모델 → 사내 반입(Gallery) → 엔진으로 실행(Runtime) → 표준 API로 서빙(Endpoint)"이라는 한 줄이 PAIS가 생태계를 묶는 방식입니다. 이 흐름의 구축·운영은 시리즈 ③이 본격적으로 다룹니다.
