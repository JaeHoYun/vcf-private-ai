# 07 — 관측성·운영

> 기반 버전은 [README 버전 기준 문서](../README.md#기반-버전-source-of-truth)를 참조하세요.

추론 API를 프로덕션에서 운영하려면 가용성·성능·비용을 지속적으로 관찰해야 합니다. 이 문서는 추론 API를 프로덕션에서 **무엇으로 관찰하고, 무엇을 조심하는가**를 정리합니다.

---

## 7.1 토큰 사용량 — 응답에 들어 있는 1차 신호

모든 추론 응답의 `usage` 필드가 가장 기본적인 운영 신호입니다. 별도 도구 없이 바로 쓸 수 있습니다.

| 필드 | 의미 | 운영 활용 |
|------|------|----------|
| `prompt_tokens` | 입력 토큰 수 | 프롬프트·컨텍스트가 과도하게 커지는지 감지(RAG 청크 과다 주입 등) |
| `completion_tokens` | 생성 토큰 수 | 응답 길이·비용 추적 |
| `total_tokens` | 합계 | 사용량 집계·내부 과금(쇼백) 기준 |

> 토큰 사용량을 **호출 단위로 로깅**해 두면, 어느 앱·사용자·에이전트가 GPU를 많이 쓰는지 사후 분석이 가능합니다. 사내 추론은 종량 청구서가 오지 않으므로, **쇼백(showback)을 직접 만들어야** 비용 책임이 보입니다.

## 7.2 LLM 트레이싱 (OpenTelemetry)

PAIS 2.1은 **OpenTelemetry(OTel) 기반 LLM 트레이싱**을 제공합니다. 에이전트 호출은 검색·도구 호출·추론 등 여러 단계를 거치므로, 단계별 추적이 디버깅·성능·비용 분석의 핵심입니다.

| 추적 대상 | 확인 가능한 것 |
|----------|--------------|
| LLM 호출 | 토큰 사용량, 지연, 캐시 활용률 |
| RAG 검색 | 검색된 청크, 유사도(Similarity) 점수 |
| MCP 도구 호출 | 호출된 도구·인자·결과·소요 시간 |
| 전체 트레이스 | 한 요청의 end-to-end 단계별 타임라인 |

> **알려진 이슈(PAIS 2.1):** 일부 환경에서 PAIS의 LLM 트레이스가 OpenTelemetry Collector에 표시되지 않을 수 있다고 릴리스 노트에 기재되어 있습니다. 관측성 구성 시 **트레이스 수신을 반드시 검증**하고, 적용 직전 릴리스 노트에서 해당 이슈의 상태를 확인하시기 바랍니다.

## 7.3 Grafana 기반 PAIS 관측성 — 모델 health·quality·behavior

PAIS 2.1은 OTel 트레이싱과 더불어 **관측성 프레임워크로 AI 메트릭 대시보드**를 제공합니다. 에이전트에 사용되는 모델의 **health(가용성)·quality(품질)·behavior(동작)** 을 추적하도록 설계되어 있으며, 메트릭 시각화는 **Grafana**에 띄우는 것을 전제로 합니다. (VCF 9.1 블로그: "These AI Metrics dashboards require enterprises to deploy Grafana".)

| 관측 축 | 무엇을 보는가 | 운영 판단 |
|---------|--------------|----------|
| **Health** | 모델 엔드포인트 가용성·지연·복제본 상태 | 다운/포화 감지, 스케일·롤백 트리거 |
| **Quality** | 검색 품질(유사도 분포)·응답 품질 신호 | RAG 청크/모델 교체의 회귀 감지 → [7.5](#75-모델-버전엔드포인트-관리) |
| **Behavior** | 토큰 사용 패턴·도구 호출 패턴·이상 동작 | 비정상 사용·비용 급증·프롬프트 인젝션 징후 추적 |

**관측성 데이터 흐름(개념)**

```
PAIS(추론 엔진·KB·에이전트)
   │  메트릭/트레이스 (OTel)
   ▼
OpenTelemetry Collector ──▶ 메트릭 백엔드(Prometheus 등)
                                   │
                                   ▼
                              Grafana 대시보드
                        (모델 health · quality · behavior)
```

> **전제 조건** — AI 메트릭 대시보드는 조직이 **Grafana를 직접 배포**해야 동작합니다. 즉 관측성은 "켜져 있는" 기능이 아니라 **배포·연결해야 하는** 구성입니다. 정확한 메트릭 항목·대시보드 구성은 적용 직전 [PAIS 릴리스 노트](https://techdocs.broadcom.com/us/en/vmware-cis/private-ai/foundation-with-nvidia/9-0/private-ai-release-notes/vmware-private-ai-services-release-notes.html)와 [VCF 9.1 블로그](https://blogs.vmware.com/cloud-foundation/2026/05/05/streamline-simplify-and-protect-all-your-ai-workloads-with-vcf-9-1/)로 확인하시기 바랍니다.

## 7.4 3계층 모니터링

추론 API 운영은 한 계층만 봐서는 안 됩니다. 문제의 원인이 어느 층인지 분리해야 합니다.

```
앱 레벨 (개발팀)
 • 요청 수 / 응답시간(P50·P95·P99) · 에러율(4xx·5xx) · 동시 사용자
AI 서비스 레벨 (PAIS)
 • Model Endpoint 응답시간 · 토큰 사용량 · Agent 호출 수 · 검색 품질(score 분포)
인프라 레벨 (플랫폼팀)
 • GPU 사용률·메모리 · 복제본 상태·재시작 · 네임스페이스 쿼터
```

> 증상-원인 분리 예: "응답이 느리다" → 앱 레벨 P95가 튀는데 PAIS Endpoint 응답시간은 정상이면 **앱/네트워크 문제**, PAIS Endpoint 자체가 느리면 **복제본 부족/GPU 포화**(인프라 레벨 확인). 관측성이 없으면 이 분리가 불가능합니다.

## 7.5 모델 버전·엔드포인트 관리

사내 추론에서는 모델 교체 시기와 버전을 조직이 직접 결정합니다. 다만 교체 절차를 잘 잡지 않으면 앱 품질이 의도치 않게 바뀔 수 있습니다.

| 항목 | 권장 |
|------|------|
| **모델 식별** | 앱은 `model` 이름을 환경설정(ConfigMap 등)으로 분리. 코드에 하드코딩 금지 |
| **버전 전환** | 새 모델 엔드포인트를 띄워 병행 검증(섀도/카나리) 후 전환. 한 번에 갈아끼우지 않기 |
| **호환성** | 모델 교체 시 프롬프트·출력 형식이 달라질 수 있음. 회귀 테스트(품질 평가) 필수 |
| **롤백** | 이전 모델 엔드포인트를 일정 기간 유지해 빠른 롤백 경로 확보 |

> AI 앱의 회귀는 "에러"가 아니라 **"답의 품질 저하"** 로 나타납니다. 단위 테스트로는 안 잡힙니다. 모델 교체 전후로 대표 질문 세트에 대한 **정확도·관련성 평가**를 돌리세요.

## 7.6 비용 — GPU가 핵심

| 비용 요소 | 비중 | 최적화 |
|----------|------|--------|
| **GPU(추론 복제본)** | 가장 큼 | 적정 모델 크기 선택, 자동 스케일링(최소 복제본), 유휴 시 축소 |
| 임베딩·경량 추론 | 중간 | CPU 추론 엔진 활용 가능(Infinity·llama.cpp). 9.1에선 completion도 CPU(llama.cpp)로 가능 → 경량·PoC는 GPU 절약 |
| 스토리지 | 작음 | Harbor 모델 이미지·벡터 데이터 |

> **유휴 복제본도 GPU를 점유하면 비용입니다.** 단, 복제본을 0으로 내리면 첫 응답이 느려집니다(Cold Start). 첫 응답 지연(TTFT)이 중요한 사용자 대면 서비스는 **최소 복제본 ≥ 1**, 배치·내부 작업은 더 공격적으로 축소하는 식으로 워크로드별로 다르게 잡으세요. 상세 스케일링·HA/DR은 형제 가이드 [06](../../01-infra/docs/06-production.md) 참조.
