# 03 — 설계 결정: 컴퓨트·GPU·VKS 토폴로지

[← 목차로](../README.md)

컴퓨트 계층은 Private AI 설계에서 가장 먼저, 가장 되돌리기 어려운 결정들이 몰리는 곳입니다. 토폴로지와 GPU 공유 방식은 설치 시점에 사실상 확정되며 사후 변경 비용이 큽니다. 이 문서는 네 개의 설계 결정을 다룹니다. 각 결정은 **경로 → 비교표 → 상황별 권고 → 함정** 순서로 정리합니다.

> 본 문서의 수치·동작은 작성 시점(2026-06) VCF 9.1 / PAIF 9.1 / PAIS 2.1 기준이며, 적용 전 공식 문서로 재확인하시기 바랍니다.

---

## 3.1 결정: VCF 토폴로지 — 표준(standard)을 기본으로

**무엇을 정하나** — 관리 도메인과 GPU 가속 워크로드 도메인을 분리할지(표준), 한 클러스터에 합칠지(통합). Private AI에서는 **표준이 기본**이며 통합은 권고하지 않습니다.

**기본 설계 — 표준(standard)**: 관리 도메인과 GPU 가속 워크로드 도메인을 물리적으로 분리합니다. 이 구성을 출발점으로 삼는 이유는 GPU 워크로드의 독립 확장·수명주기 관리, 관리 컴포넌트와의 자원 격리, 테넌시·보안 경계 확보, VKS Supervisor와 GPU 노드의 안정적 적재가 모두 가능하기 때문입니다.

**통합(consolidated)은 권고하지 않습니다.** 관리와 워크로드를 단일 클러스터에 합치는 통합 구성은 기술적으로 가능하지만, 이는 **VCF 최소 구성을 위한 형태**이지 Private AI 워크로드에는 맞지 않습니다. 사유:

- 관리 컴포넌트와 GPU 워크로드가 같은 클러스터 자원을 경합해 AI 성능·안정성을 해칩니다.
- GPU 워크로드 도메인을 관리와 독립적으로 확장·업그레이드할 수 없습니다.
- VKS Supervisor와 GPU 노드를 안정적으로 적재하기에는 단일 클러스터 자원이 빠듯합니다.
- 테넌시·보안 격리가 약합니다.
- 프로덕션 전환 시 재배포에 가까운 작업이 필요합니다(토폴로지는 설치 시점에 확정).

> 예외 — 버려도 되는 순수 기능 검증용 랩에 한해 통합을 쓸 수 있으나, 그 환경을 그대로 프로덕션으로 키우지 마십시오.

## 3.2 결정: GPU 공유 방식 — MIG vs 타임슬라이싱(vGPU) vs 풀 패스스루(DirectPath)

**무엇을 정하나** — 한 물리 GPU를 워크로드에 어떻게 나눌지. 활용률·격리·성능 예측성의 삼각 트레이드오프입니다.

- **경로 A — MIG(Multi-Instance GPU)**: 지원 GPU(예: A100·H100·H200급)를 하드웨어 수준으로 분할. 인스턴스 간 강한 격리와 예측 가능한 성능, 단 분할 입자도는 고정. ESXi 호스트마다 MIG 활성화 필요.
- **경로 B — 타임슬라이싱(vGPU time-share)**: 시간 분할로 여러 VM이 GPU를 공유. 유연하고 활용률이 높지만 격리·QoS가 약하고 메모리 경합이 발생. NVIDIA AI Enterprise(NVAIE) 라이선스 필요.
- **경로 C — 풀 패스스루(DirectPath I/O)**: VM 하나가 물리 GPU 전체(메모리·코어·NVLink 대역)를 독점. 최고 성능, 공유 불가, vMotion·HA 제약.

| 차원 | MIG | 타임슬라이싱 | 풀 패스스루 |
|------|-----|-------------|------------|
| 격리 | 강(하드웨어) | 약(시간분할) | 완전(독점) |
| 성능 예측성 | 높음 | 낮음(경합) | 최고 |
| 활용률 | 중(고정 입자) | 높음 | 낮음(독점) |
| vMotion·HA | 제한 | 일부 지원 | 불가 |
| 적합 워크로드 | 멀티테넌트 추론·다수 소형 모델 | dev·가변 부하 | 대형 학습·단일 대형 모델 |

**상황별 권고**
- 멀티테넌트 추론, 다수의 소형·중형 모델을 안정적으로 → **MIG**
- 개발·실험 환경, 부하가 들쭉날쭉하고 활용률이 우선 → **타임슬라이싱**
- 대형 모델 학습/파인튜닝, 단일 워크로드 최고 성능 → **풀 패스스루**

**함정** — 풀 패스스루 VM은 vMotion이 되지 않아 호스트 유지보수·HA 설계에 직접 영향을 줍니다. MIG는 지원 GPU가 한정되므로 하드웨어 선정과 함께 결정해야 합니다.

## 3.3 결정: 서빙 런타임 배치 — VKS(컨테이너) vs DLVM(딥러닝 VM)

**무엇을 정하나** — 모델 서빙·학습 워크로드를 쿠버네티스(VKS) 위 컨테이너로 올릴지, 딥러닝 VM(DLVM)에 직접 둘지.

- **경로 A — VKS(vSphere Kubernetes Service)**: 선언형 배포·오토스케일·롤아웃을 갖춘 쿠버네티스 기반. PAIS Model Runtime의 실행 기반이며 다수 모델·멀티테넌시에 유리. 쿠버네티스 운영 스킬과 일정 규모(프로덕션 GPU 노드 최소 3대급)가 전제.
- **경로 B — DLVM(Deep Learning VM)**: 인기 AI/ML 런타임이 사전 구성된 VM 이미지(콘텐츠 라이브러리)를 바로 배포. 단발 실험·단일 워크로드에 빠르고, 쿠버네티스가 불필요. 스케일·수명주기 관리는 수동.

| 차원 | VKS | DLVM |
|------|-----|------|
| 확장성·자동화 | 높음(선언형·오토스케일) | 낮음(수동) |
| 운영 복잡도 | 높음(K8s) | 낮음 |
| 멀티모델·멀티테넌시 | 강함 | 약함 |
| 최소 규모 | 큼 | 작음(단일 VM) |
| PAIS 연계 | 직접(Model Runtime 기반) | 간접 |
| 적합 | 프로덕션 서빙 | PoC·데이터과학 실험 |

**상황별 권고**
- 프로덕션 서빙, 다수 모델, 자동화·오토스케일 → **VKS**
- PoC·실험, 단일 모델, 빠른 시작·낮은 운영 부담 → **DLVM**

## 3.4 결정: 서빙 제공 방식 — PAIS Model Runtime vs NIM 직접 vs 자가 vLLM

**무엇을 정하나** — 모델을 어떤 추론 스택으로 노출할지. 전형적인 빌드 vs 바이(build vs buy) 결정입니다.

- **경로 A — PAIS Model Runtime(관리형)**: VMware가 관리하는 서빙 계층으로 vLLM·Infinity·llama.cpp 추론 엔진을 내장하고 Model Gallery와 OpenAI 호환 API를 제공. 운영 부담이 가장 적고 플랫폼 통합도가 높음.
- **경로 B — NIM 직접(NVIDIA Inference Microservices)**: NVIDIA가 최적화한 추론 마이크로서비스. 검증된 모델에서 최고 성능과 기업 지원, 단 NVAIE 라이선스와 NGC(NVIDIA GPU Cloud, NVIDIA의 컨테이너·모델 배포 카탈로그) 의존.
- **경로 C — 자가 vLLM(직접 운영)**: vLLM 등을 직접 컨테이너로 운영. 최신 오픈 모델·커스텀에 최대 유연성, 운영·지원은 자체 부담.

| 차원 | PAIS Model Runtime | NIM 직접 | 자가 vLLM |
|------|-------------------|----------|-----------|
| 운영 부담 | 낮음(관리형) | 중 | 높음 |
| 성능 | 양호 | 최고(최적화) | 모델·튜닝 의존 |
| 모델 범위·최신성 | 검증 카탈로그 | NIM 카탈로그 | 최신 오픈소스(OSS) 자유 |
| 지원 | VMware | NVIDIA 기업 지원 | 커뮤니티·자체 |
| 라이선스·통합 | 플랫폼 내장 | NVAIE | 자유·자가 통합 |

**상황별 권고**
- 표준 운영, 빠른 도입, 적은 손길 → **PAIS Model Runtime**
- 검증된 모델로 최고 성능과 기업 지원 → **NIM 직접**
- 최신 OSS·커스텀 모델, 연구·특수 요구 → **자가 vLLM**
