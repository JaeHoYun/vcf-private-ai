# 03 — GPU 101: 인프라 엔지니어를 위한 GPU 기초

이 문서는 가상화·스토리지·네트워크에 익숙한 인프라 엔지니어가 GPU를 자신이 아는 개념에 빗대어 잡는 입문서입니다. "왜 AI에는 CPU가 아니라 GPU인가"에서 시작해, GPU가 물리 서버에서 컨테이너까지 어떻게 노출되는지를 직관 수준으로 잇습니다. 분할·할당의 정확한 매트릭스와 구성 절차는 본편 ①이 다루며, 이 문서는 그 본편을 읽기 위한 어휘를 잡는 데 집중합니다.

비유는 직관을 잡기 위한 것이며, 정확한 정의·수치·절차는 본편과 각 벤더의 공식 문서를 따릅니다.

## 3.1 왜 AI에는 GPU인가

CPU는 소수의 강력한 코어로 복잡한 작업을 빠르게 순차 처리하도록 설계되었습니다. 반면 GPU(Graphics Processing Unit)는 단순한 코어를 수천 개 담아, 같은 연산을 방대한 데이터에 **동시에** 적용하는 데 특화되어 있습니다.

멘탈모델로 보면, CPU는 다재다능한 숙련공 몇 명이고 GPU는 단순 작업을 동시에 수행하는 수천 명의 일꾼입니다. 한 명 한 명은 단순하지만, 똑같은 일을 한꺼번에 처리해야 할 때는 압도적으로 빠릅니다.

AI 모델의 추론과 학습은 본질적으로 거대한 **행렬 곱셈**의 반복입니다(§1.1의 "거대한 함수"가 실제로 하는 일). 이 연산은 수많은 곱셈·덧셈을 동시에 수행할 수 있어 GPU의 병렬 구조와 정확히 맞아떨어집니다. 그래서 같은 모델이라도 CPU로는 실용적이지 않은 속도가, GPU에서는 실시간에 가까워집니다.

## 3.2 GPU 메모리(VRAM) — 가장 결정적인 제약

GPU에는 자체 메모리가 붙어 있습니다. 이를 **VRAM**(Video RAM) 또는 GPU 메모리라고 부릅니다. 인프라 관점에서 GPU를 다룰 때 가장 먼저 마주치는 제약이 바로 이 VRAM 용량입니다.

이유는 단순합니다. 모델이 추론을 하려면 **가중치 전체가 VRAM에 올라가 있어야** 합니다(§1.1.1). 70B 모델은 16비트 정밀도 기준 대략 수십 GB의 VRAM을 차지합니다. 여기에 추론 중 생성되는 중간 데이터(KV 캐시 등)와 동시 처리량(배치)에 따른 추가 메모리가 더 필요합니다.

VM에 빗대면, VRAM은 "이 GPU에 올릴 수 있는 워크로드의 메모리 한도"입니다. 호스트 RAM이 부족하면 VM이 안 뜨듯, VRAM이 부족하면 모델이 아예 안 올라갑니다. 그래서 §1.1.3의 양자화(정밀도를 낮춰 메모리를 줄이는 손잡이)가 사이징에서 그렇게 자주 등장하는 것입니다.

핵심은 GPU 선택이 연산 속도보다 **VRAM 용량에서 먼저 갈린다**는 점입니다. "이 모델이 이 GPU에 올라가는가"가 1차 관문이고, "얼마나 빠른가"는 그다음 질문입니다. 정확한 용량 산정은 시리즈 ⑥(사이징)이 다룹니다.

## 3.3 한 장으로 부족할 때 — 여러 GPU로 나누기

모델이 GPU 한 장의 VRAM에 들어가지 않거나, 처리량을 더 끌어올려야 할 때는 여러 장의 GPU를 함께 씁니다. 이때 GPU들이 서로 데이터를 주고받는 속도가 병목이 되므로, GPU 간 연결 방식이 중요해집니다.

- **한 노드 안의 여러 GPU** — 기본 연결은 서버의 표준 버스인 **PCIe**입니다. 여기에 더해 일부 GPU·서버 조합은 **NVLink** 같은 전용 고속 인터커넥트를 추가로 제공해, PCIe보다 훨씬 빠르게 GPU끼리 직접 통신합니다. 즉 NVLink는 "있으면 더 빠른" 선택지이지 멀티 GPU의 전제는 아닙니다. NVLink가 없는 GPU도 PCIe로 여러 장을 묶어 씁니다.
- **여러 노드에 걸친 GPU** — 서버를 넘어서면 고속 네트워크로 GPU들을 잇습니다. 지연을 줄이기 위해 **RDMA**(InfiniBand나 RoCE 같은 기술)를 흔히 씁니다. 노드 경계를 넘는 통신은 노드 안보다 느리므로, 설계에서 신중히 다뤄야 합니다.

모델을 여러 GPU에 나눠 올리는 방식(텐서 병렬·파이프라인 병렬 등)에는 여러 갈래가 있지만, 입문 단계에서는 "한 장에 안 들어가면 여러 장에 쪼개 올리고, 그때 GPU 간 연결 속도가 성능을 좌우한다"는 직관이면 충분합니다. 구체적인 토폴로지와 인터커넥트 요구사항은 본편 ①이 다룹니다.

## 3.4 GPU를 전용으로 줄까, 나눠 줄까

한 장의 GPU를 워크로드 하나가 통째로 쓰기도 하지만, 작은 워크로드 여러 개가 한 GPU를 나눠 쓰는 편이 효율적일 때도 많습니다. 그래서 GPU를 워크로드에 붙이는 방식에는 크게 세 갈래가 있습니다. 여기서는 직관만 잡고, 어떤 모드를 언제 쓰는지의 결정 가이드와 호환성 매트릭스는 본편 ①(특히 GPUaaS·GPU enablement 문서)에 맡깁니다.

- **Passthrough(전용)** — GPU 한 장을 통째로 VM 하나에 직접 붙입니다. 전용 주차 공간을 한 대에 통째로 내주는 것과 같습니다. 격리와 성능은 확실하지만, 한 장을 한 워크로드가 독점합니다.
- **vGPU(분할 공유)** — 한 장의 GPU를 소프트웨어로 여러 몫으로 나눠 여러 VM에 나눠 줍니다. 한 주차장을 칸막이로 나눠 여러 대가 쓰는 것에 가깝습니다.
- **MIG(하드웨어 격리 분할)** — GPU를 하드웨어 수준에서 독립된 인스턴스 여러 개로 쪼갭니다. 칸막이를 콘크리트 벽으로 세운 분할이라고 보면 됩니다. 몫끼리 더 강하게 격리됩니다.

멘탈모델은 "GPU를 통째로 줄 것인가(passthrough), 소프트웨어로 나눌 것인가(vGPU), 하드웨어로 갈라 줄 것인가(MIG)"입니다. 이 선택이 격리 수준·비용 효율·라이선스에 영향을 주며, 그 균형을 잡는 것이 ①의 핵심 주제 중 하나입니다.

## 3.5 GPU를 들이는 데 따르는 물리 제약 — 전력·발열·공간

GPU는 일반 서버 부품과 다른 시설 부담을 동반합니다. 인프라 담당자가 GPU 도입에서 가장 먼저 점검하게 되는 현실적 제약이기도 합니다.

- **전력** — 고성능 GPU 한 장이 수백 W를 소비하며, GPU를 여러 장 꽂은 서버 한 대는 일반 서버의 몇 배에 달하는 전력을 끌어씁니다. 랙당 전력 예산(power budget)이 "이 랙에 GPU 서버를 몇 대 꽂을 수 있는가"를 먼저 제한하는 경우가 많습니다.
- **발열·냉각** — 전력을 많이 쓰는 만큼 열도 많이 냅니다. 기존 공조로 감당이 안 되면 랙을 듬성하게 채우거나 별도 냉각(경우에 따라 수냉)을 검토해야 합니다.
- **물리 폼팩터** — GPU 카드는 크고 무거우며 전원 요구가 높아 아무 서버에나 꽂히지 않습니다. GPU를 지원하는 서버 모델·슬롯·전원 커넥터가 전제됩니다.

멘탈모델로 보면, GPU 서버는 "전기를 많이 먹고 열을 많이 내는 무거운 워크로드"입니다. 가상화 용량(CPU·RAM)만 보던 시각에 전력·냉각·랙 공간이라는 축이 하나 더 붙는 셈입니다. 정확한 전력·냉각 산정과 데이터센터 설계는 시리즈 ⑥(사이징)·①(인프라)이 다루며, 이 입문서는 "이런 축이 새로 생긴다"는 점까지만 짚습니다.

## 3.6 두 계층 노출 경로 — 물리 GPU가 컨테이너에 닿기까지

§2.3.3에서 짚은 인프라의 까다로운 지점을 한 겹 더 풀어 보겠습니다. 물리 GPU가 모델에 쓰이려면, 다음 수직 경로를 모두 통과해야 합니다.

```
물리 GPU (하드웨어)
   │  하이퍼바이저가 GPU를 인식 (드라이버/구성)
   ▼
하이퍼바이저 (vSphere)
   │  GPU를 VM(쿠버네티스 노드)에 할당 (passthrough/vGPU/MIG)
   ▼
노드 (VM)
   │  GPU Operator가 노드에 드라이버·런타임 준비 (§2.4.2)
   ▼
컨테이너 (Pod)
   │  모델이 비로소 GPU를 사용
   ▼
모델 서빙
```

각 단계가 하나라도 어긋나면 위쪽 단계가 멀쩡해도 모델은 GPU를 보지 못합니다. 그래서 GPU enablement는 "어느 계층에서 GPU가 보이는가"를 단계별로 확인하는 작업이 됩니다. 이 경로의 버전 호환성·구성 절차·트러블슈팅은 본편 ①이 핸즈온으로 다루며, 이 입문서는 "이런 수직 경로가 있다"는 지도까지만 그립니다.

이 경로 위에서 도는 소프트웨어 묶음(드라이버·CUDA·서빙 엔진·생태계 도구)이 무엇인지는 다음 문서에서 이어집니다.
