# 00 — 추론 서빙은 어떻게 동작하나 (기초)

> 기반 버전은 [README 버전 기준 문서](../README.md#기반-버전-source-of-truth)를 참조하세요.

이 가이드의 본문(01~08)은 모델을 API로 **서빙하는 방법**을 다룹니다. 그 전에 이 문서는 그 API 뒤에서 무슨 일이 일어나는지 — 왜 GPU가 필요하고, 양자화·추론 엔진·동시성이 무엇이며, 왜 같은 코드로 CPU와 GPU를 오갈 수 있는지 — 를 **개념 수준에서** 짚습니다. 정밀한 수식·사이징 수치는 [⑥ 사이징·비용](../../06-sizing-cost/README.md)에, 인프라 구축은 [① 인프라](../../01-infra/README.md)에 위임하고, 여기서는 "왜 이렇게 되는가"의 직관만 세웁니다.

> 이미 LLM 서빙 내부에 익숙하시다면 [01 왜 사내 추론 API인가](01-why-serving-api.md)로 건너뛰어도 됩니다.

---

## 0.1 왜 추론에 GPU인가

LLM이 답을 만드는 과정은, 안을 들여다보면 결국 **어마어마하게 많은 곱셈과 덧셈**입니다. 토큰(단어 조각) 하나를 만들 때마다 수십억 개의 숫자(파라미터)를 곱하고 더합니다.

- **CPU**는 똑똑한 만능 일꾼이 몇 명(수~수십 코어) 있는 구조입니다.
- **GPU**는 단순 계산만 하는 일꾼이 수천 명 있는 구조입니다.
- 단순한 곱셈을 **동시에 아주 많이** 해야 하는 일에는 수천 명이 압도적으로 유리합니다.

비유하면, 똑같은 덧셈 문제 100만 개를 풀 때 천재 4명(CPU)보다 평범한 계산원 5천 명(GPU)이 훨씬 빨리 끝냅니다. 그래서 **추론 = GPU** 라는 공식이 생겼습니다. 토큰을 한 개씩 이어 만드는 생성(디코딩) 단계는 특히 메모리에서 가중치를 읽어오는 속도(메모리 대역폭)에 민감해, 대역폭이 큰 GPU가 유리합니다.

---

## 0.2 무엇이 GPU 메모리를 먹나

GPU 메모리(VRAM)에는 두 가지가 올라갑니다.

- **모델 가중치(고정)** — 모델 크기 = 파라미터 수 × 파라미터당 바이트 수. 예) 8B(80억) 모델을 FP16(16비트)로 두면 약 16GB, 70B는 약 140GB로 단일 GPU 한 장을 넘깁니다.
- **KV 캐시(가변)** — 생성 중인 각 요청의 "지금까지의 문맥"을 들고 있는 메모리. 동시 요청이 많고 컨텍스트가 길수록 커집니다. ("토큰"·"컨텍스트"가 생소하면 §0.8을 먼저 보십시오.)

이 둘의 합이 VRAM을 넘으면 모델이 안 올라가거나 동시 처리 수가 줄어듭니다. **그래서 "모델 크기 선택"이 가장 큰 결정**입니다. 가중치·KV 캐시의 정밀한 산정 수식과 예제는 [⑥ GPU 사이징](../../06-sizing-cost/docs/02-gpu-sizing.md) §2.2·§2.3에 있습니다.

---

## 0.3 양자화 — 모델을 가볍게

**양자화(quantization)** 는 숫자의 정밀도를 낮춰 모델을 압축하는 기법입니다. 예를 들어 FP16(16비트)을 INT4(4비트)로 바꾸면 용량이 약 1/4로 줄어, 8B 모델이 16GB에서 약 4~5GB까지 작아집니다. 이 정도면 일반 서버의 CPU 메모리에도 들어가 **GPU 없이 CPU로도 추론**할 수 있습니다.

비유하면 고해상도 사진을 적당히 압축하는 것과 같습니다. 용량이 확 줄고 다루기 쉬워지되, 화질이 약간 흐려집니다(품질을 조금 내주는 거래). CPU 추론 엔진(0.4)이 쓰는 **GGUF** 형식이 바로 이렇게 양자화된 단일 파일 모델 형식입니다(GGUF 표기는 [02 §2.5](02-serving-api-architecture.md)). 양자화의 절감률·품질 트레이드오프 정량은 [⑥ GPU 사이징](../../06-sizing-cost/docs/02-gpu-sizing.md) §2.4를 보십시오.

---

## 0.4 세 가지 추론 엔진 — 언제 무엇을

PAIS의 Model Runtime 안에는 성격이 다른 추론 엔진 셋이 들어 있습니다. 같은 일을 하는 게 아니라 **각자 잘하는 영역이 다릅니다.**

- **vLLM (GPU)** — 대규모·실시간, 동시 사용자가 많은 운영(prod) 추론의 정석.
- **llama.cpp (CPU)** — 소규모·테스트·PoC·가벼운 배치. GPU 없이 싸게 돌립니다(0.3의 GGUF).
- **Infinity (CPU)** — 임베딩(검색용 벡터) 전용.

엔진별 가속기·API 식별(`model_engine`)을 담은 비교표는 [02 §2.5](02-serving-api-architecture.md)에 있습니다. 이 문서는 "그래서 언제 무엇을 고르나"의 직관만 둡니다 — **실시간·고동시 → vLLM, 비용·테스트·저부하 → llama.cpp, 검색 벡터 생성 → Infinity.**

---

## 0.5 같은 API, 다른 엔진 — 표준 추상화

가장 중요한 연결 고리입니다. 앱은 뒤에서 어떤 엔진이 어떤 가속기(GPU/CPU)로 돌든 **똑같은 방식으로 호출**합니다. 이것이 PAIS의 **OpenAI 호환 API**입니다.

비유하면 **벽의 전기 콘센트**입니다. 그 전기가 태양광에서 왔든 화력에서 왔든 플러그 구멍 모양은 같아서, 발전 방식을 바꿔도 가전제품은 그대로 꽂아 씁니다. 마찬가지로 엔진을 vLLM에서 llama.cpp로 바꿔도 **앱 코드는 한 줄도 안 바꿔도 됩니다.** 이 추상화가 왜 중요한지는 [01 §1.2](01-why-serving-api.md)에서, 실제 호출 방식은 [03 OpenAI 호환 엔드포인트](03-openai-compatible-endpoints.md)에서 다룹니다.

---

## 0.6 vLLM은 어떻게 동시에 많은 요청을 처리하나

운영 추론(vLLM, GPU)의 처리량을 끌어올리는 핵심은 GPU를 **놀리지 않게** 만드는 두 가지 기법입니다.

**(1) 연속 배칭(continuous batching)** — 요청이 끝나는 즉시 그 자리에 대기 중인 새 요청을 밀어 넣어 GPU를 항상 꽉 채웁니다.

```
정적 배칭 — 한 배치가 다 끝나야 다음 시작
  R1  [====............]   먼저 끝나도 빈칸(.)이 논다
  R2  [========........]
  R3  [================]   가장 긴 요청이 배치 길이를 정함
  R4  [======..........]
                       |-- 여기까지 GPU가 논다(낭비)

연속 배칭 — 빈칸에 즉시 새 요청 투입
  R1  [====|새 요청====]   끝나는 즉시 다음 요청이 그 자리에
  R2  [========|새 요청]
  R3  [================]
  R4  [======|새 요청==]
      빈칸이 없다 → 같은 GPU로 더 많은 처리량
```

정거장마다 모두 내릴 때까지 기다리는 버스가 아니라, 자리가 비는 즉시 다음 승객을 태우는 버스입니다.

**(2) PagedAttention** — KV 캐시(0.2)를 운영체제의 가상 메모리 페이징처럼 **작은 블록으로 쪼개 필요한 만큼만** 할당합니다. 요청마다 최대 길이로 미리 자리를 잡아두던 낭비가 사라져, 같은 VRAM에 훨씬 많은 동시 요청을 욱여넣습니다. 큰 회의실을 한 명이 통째로 예약하는 대신 칸막이로 나눠 들어오는 만큼만 내주는 것과 같습니다.

그래서 **동시 처리의 진짜 천장은 "가중치를 빼고 남은 VRAM ÷ 요청당 KV 캐시"** 입니다. 이 정량 계산(KV 캐시 산정·최대 동시성 역산)은 [⑥ GPU 사이징](../../06-sizing-cost/docs/02-gpu-sizing.md) §2.3·§2.9가 수식과 예제로 다룹니다.

> 연속 배칭·PagedAttention은 vLLM 엔진의 동작 방식입니다. 엔진 버전·세부 동작은 릴리스마다 바뀌므로, 적용 직전 엔진 버전 기준선([README 버전 기준 문서](../README.md#기반-버전-source-of-truth))과 공식 문서로 확인하시기 바랍니다.

## 0.7 개발은 CPU, 운영은 GPU

0.4~0.6을 합치면 실무 패턴이 자연스럽게 나옵니다.

- **개발·테스트**: GPU 없는 CPU 환경에 작은 양자화 모델을 llama.cpp로 띄워, 여러 개발자가 "흐름이 도느냐"를 싸게 검증합니다.
- **운영**: 진짜 트래픽에는 GPU에 vLLM + 제대로 된 모델을 올립니다.
- **전환이 매끄러운 이유**: API가 같으니(0.5), 개발에서 운영으로 갈 때 **모델·엔진만 바꾸고 앱 코드는 그대로** 둡니다.

단, CPU(llama.cpp)는 "동작이 맞는지" 검증에 적합하고, "얼마나 빠른가·몇 명을 버티나"의 성능·부하는 반드시 운영과 같은 GPU 구성에서 실측해야 합니다. DLVM(개발)→Model Endpoint(운영) 라이프사이클은 [① 개발 시나리오](../../01-infra/docs/04-dev-scenarios.md) §4.1을 참조하십시오.

## 0.8 토큰과 컨텍스트 윈도우

모델은 글자가 아니라 **토큰(token)** 단위로 텍스트를 다룹니다. 토큰은 자주 등장하는 글자·어절 묶음입니다. 대략 영어는 4글자에 1토큰, 한글은 1~3글자에 1토큰 정도이며(근사), 토크나이저(BPE 등)가 이 분할을 합니다.

왜 중요하냐면, 앞에서 본 **KV 캐시(§0.2)·메모리·비용이 전부 토큰 수를 기준**으로 계산되기 때문입니다. "3KB짜리 텍스트가 왜 수백~수천 토큰인가"를 토큰 개념 없이는 이해할 수 없습니다.

**컨텍스트 윈도우(context window)** 는 모델이 한 번에 받아들일 수 있는 **토큰 상한**입니다(예: 8K·32K·128K). 모델이 훈련 때 본 최대 길이라, 이를 넘기면 성능이 떨어지거나 거부됩니다. RAG에서는 이 한 줄이 숨은 제약이 됩니다.

```
[ 시스템 프롬프트 ] + [ 검색 근거 청크 ] + [ 사용자 질문 ] + [ 출력 여유 ]  ≤  컨텍스트 윈도우
```

그래서 **청크 크기·검색 top-k·모델 선택이 모두 윈도우 한계에 묶입니다** — 윈도우가 8K인 모델로는 근거를 적게밖에 못 담고, 128K 모델이라야 긴 근거를 넣습니다. 토큰당 KV 캐시·동시성의 정량 계산은 [⑥ GPU 사이징](../../06-sizing-cost/docs/02-gpu-sizing.md) §2.3에 있습니다.

## 0.9 모델 타입·포맷·샘플링

서빙할 모델을 고를 때 알아 둘 세 가지 기초입니다.

- **베이스 모델 vs Instruct/Chat 모델** — 베이스(base)는 "다음 토큰 예측"만 학습한 원형이고, Instruct/Chat은 그 위에 "지시를 따르도록" 추가 튜닝한 것입니다. 대화·지시형 앱에는 **Instruct/Chat 모델**을 씁니다. 요즘 오픈웨이트 모델은 대부분 Instruct 변형을 함께 제공합니다.
- **파운데이션 모델·오픈웨이트** — 대규모로 사전학습된 다목적 모델을 파운데이션 모델이라 하고, 그중 가중치가 공개된 것(Llama·Mistral·Qwen 등)을 **오픈웨이트(open-weight)** 라 합니다. 가중치를 받아 **사내 GPU에서 직접 실행**할 수 있다는 점이 Private AI의 전제입니다.
- **모델 포맷** — 같은 모델도 저장 형식이 다릅니다. **safetensors**(가중치 표준 저장 형식), **GGUF**(llama.cpp용 양자화 단일 파일, §0.3), **NIM**(NVIDIA 추론 컨테이너 런타임). Model Gallery에 반입할 때 어느 형식으로 둘지 결정합니다.
- **샘플링 파라미터** — 같은 모델·같은 질문도 출력 제어값에 따라 답이 달라집니다. **temperature**(0이면 거의 항상 같은 답, 높을수록 무작위·창의), **top-p**(누적 확률 컷으로 후보 토큰 제한), **max tokens**(출력 길이 상한). 일관된 답이 필요하면 낮은 temperature, 창의가 필요하면 높입니다. 실제 호출 파라미터는 [03 OpenAI 호환 엔드포인트](03-openai-compatible-endpoints.md)에서 다룹니다.
