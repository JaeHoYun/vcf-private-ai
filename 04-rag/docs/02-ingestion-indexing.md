# 02 — 데이터 인입·인덱싱

[← 목차로](../README.md)

인덱싱 타임의 작업입니다. 검색 품질의 70%는 여기서 결정됩니다 — 잘못 쪼개고 잘못 임베딩한 데이터는 어떤 검색 알고리즘으로도 살릴 수 없습니다.

## 2.1 문서 로딩

사내 문서는 형식이 제각각입니다(PDF, Office, HTML, 위키, 티켓). 로딩 단계의 목표는 **원문 텍스트 + 메타데이터**를 정규화해 뽑는 것입니다.

- **텍스트 추출**: PDF는 레이아웃/표가 깨지기 쉬우니 표·머리말·꼬리말 처리에 주의. 스캔 PDF는 OCR 필요.
- **메타데이터 보존**: 문서 제목·출처 경로·부서·작성일·접근등급을 함께 저장합니다. 이 메타데이터가 나중에 **출처 표기**와 **접근 통제 필터링**의 근거가 됩니다.
- **접근등급 태깅**: 인덱싱 시점에 문서별 권한을 메타데이터로 박아두면, 검색 시 사용자 권한으로 필터링할 수 있습니다(07 멀티테넌트 참조).

### 2.1.1 이기종 소스 연결 패턴

PAIS의 관리형 인입은 네 종류의 커넥터(Google Drive·Confluence·SharePoint·S3)로 한정됩니다. 지원 커넥터 목록의 기준 문서는 시리즈 ②이며(교차링크: [② 05-usage-rag](../../02-vectordb/docs/05-usage-rag.md)), 본 절은 그 범위 밖을 다룹니다. 네 종류 밖의 소스(레거시 관계형 데이터베이스, 파일 셰어, 사내 자체 시스템, 미지원 형식)는 별도의 커스텀 인제스트 파이프라인이 필요합니다.

공통 파이프라인은 단계별 순차 변환 구조로 구성합니다: 추출(extract) → 정규화·정제(normalize/clean) → 청킹(chunk) → 임베딩(embed) → 적재(load). ([Unstructured](https://unstructured.io/insights/rag-systems-best-practices-unstructured-data-pipeline), [AWS Big Data Blog](https://aws.amazon.com/blogs/big-data/build-a-rag-data-ingestion-pipeline-for-large-scale-ml-workloads/))

소스별 추출 로직은 분리하되, 하류 단계는 공통 정규화 표현으로 수렴시킵니다 — 청킹·임베딩 단계를 소스에 무관하게 재사용하기 위함입니다. 커넥터는 인증·페이지네이션·요청 속도 제한(rate limit)을 캡슐화하고, 소스 식별자(source ID)와 접근 규칙(ACL)을 콘텐츠와 함께 보존합니다. 소스 식별자와 ACL, 이 두 값이 나중에 역추적과 권한 필터의 근거가 됩니다. ([Microsoft Learn](https://learn.microsoft.com/dotnet/ai/conceptual/data-ingestion))

추출 방식은 소스 특성에 맞춰 선택합니다. 실시간·이벤트성 소스는 API 또는 웹훅으로, 대용량·정적 소스는 배치로 처리합니다. "네이티브 연결 가능 vs 커스텀 필요"의 판정 기준은 명확합니다 — 안정적인 API·스키마·권한을 제공하는 사전 제작 커넥터가 있는지로 가립니다. 그런 커넥터가 없으면 커스텀 파이프라인 대상입니다.

커스텀 커넥터는 다음 표준 형태로 구성하는 것을 권장합니다(아래 1)–5)는 본 절 안의 단계 표기일 뿐, 다른 가이드의 기호를 가리키지 않습니다).

1) 인증·연결
2) 변경 감지 커서(워터마크 또는 변경 데이터 캡처(CDC) 위치) 확보
3) 소스 추출·정규화 매퍼
4) 메타데이터·접근 규칙(ACL) 부착
5) 멱등 적재(upsert)

## 2.2 청킹 전략

**왜 쪼개나** — 문서를 통째로 임베딩하면, 그 한 벡터가 문서 안 여러 주제를 뭉뚱그린 **평균 벡터**가 됩니다. "비밀번호 재설정"만 묻는 질문이 들어와도, 100쪽 매뉴얼 전체를 대표하는 흐릿한 벡터와는 또렷이 가까워지지 않습니다. 작은 조각(chunk) 하나가 **한 주제를 또렷이 대표**할 때 비로소 검색이 정확해집니다.

**청크의 두 역할** — 청크는 (1) 검색에 걸리는 **검색 단위**이자, (2) 검색된 뒤 원문이 그대로 모델 프롬프트에 들어가는 **답변 근거 단위**입니다(03·04). 그래서 청크는 검색에 또렷이 걸릴 만큼 한 주제로 좁으면서도, 답이 될 만큼 충분한 맥락을 담아야 합니다 — 이 균형이 청킹의 핵심입니다.

| 전략 | 설명 | 적합 |
|------|------|------|
| 고정 크기 | N 토큰 단위로 자름(예: 512토큰, 10–15% 오버랩) | 균일한 일반 문서 |
| 구조 기반 | 제목/절/문단 경계로 자름 | 구조화된 정책·매뉴얼 |
| 시맨틱 | 의미 단위로 자름(임베딩 유사도 기반) | 서술형 장문 |

**어떤 전략을 고르나** — 문서의 생김새를 따릅니다.

- **구조 기반을 우선** 고려하십시오. 사내 정책·매뉴얼·FAQ처럼 제목·절·문단 경계가 곧 의미 경계인 문서가 많아, 그 경계로 자르면 한 청크가 자연히 한 주제가 됩니다.
- 경계가 흐린 **서술형 장문**(회의록·보고서 본문)은 시맨틱(의미 유사도로 자름)이나 큰 고정 크기를 씁니다.
- 형식이 제각각이라 일괄 처리해야 하면 **고정 크기 + 오버랩**이 가장 단순한 출발점입니다.

핵심 트레이드오프: **너무 작으면** 문맥이 끊겨 검색은 정확해도 답변 근거가 빈약해지고, **너무 크면** 한 청크에 여러 주제가 섞여 검색 정밀도가 떨어지고 프롬프트 토큰을 낭비합니다. 일반 문서 Q&A는 **256–512토큰 + 오버랩**에서 출발해 06의 평가로 조정하는 것을 권장합니다.

> 오버랩(인접 청크 간 중복)은 경계에 걸친 정보가 양쪽 청크에 모두 담기게 해 검색 누락을 줄입니다.

**예시** — "VPN 접속 장애 1차 점검" 절이 담긴 IT 운영 매뉴얼을 절 제목 단위(구조 기반)로 자르면 "VPN 장애 대응"이 한 청크가 됩니다. 사용자가 "VPN이 안 됩니다"라고 물으면 그 청크가 또렷이 검색되고 원문이 그대로 답변 근거로 들어갑니다. 같은 문서를 500토큰 고정으로 자르면 이 절이 앞뒤 절과 한 청크에 섞여 검색·근거가 모두 흐려질 수 있습니다.

**함정 — 표·코드·목록** — 표나 코드 블록을 토큰 수만 보고 중간에서 자르면 의미가 깨집니다(표 헤더와 행이 분리되는 등). 이런 요소는 한 덩어리로 보존하거나 구조 기반으로 처리하고, 표는 행 단위로 풀어 주는 전처리가 도움이 됩니다(레이아웃에 의존하는 문서는 2.3.1 참조).

## 2.3 임베딩

각 청크를 임베딩 모델로 벡터화합니다. PAIS Model Runtime에서 임베딩 모델을 서빙하며(③), **OpenAI 호환 임베딩 API**로 호출합니다.

```text
POST {PAIS}/compatibility/openai/v1/embeddings   # 경로는 예시 — ③/공식 API로 확인
{ "model": "<embedding-model>", "input": ["청크1", "청크2", ...] }
```

결정 지점:

- **모델 선택**: 임베딩 모델은 ③의 Model Runtime에서 서빙되는 것(예: vLLM embedding, Infinity)을 사용합니다. 사용 모델·버전은 ③과 ① 버전표를 따릅니다.
- **차원 수 일관성**: pgvector 컬럼 차원은 임베딩 모델 출력 차원과 **정확히 일치**해야 합니다. 모델을 바꾸면 전체 재인덱싱이 필요합니다.
- **질문·문서 동일 모델**: 쿼리 타임의 질문 임베딩도 **반드시 같은 모델**로 만들어야 같은 벡터 공간에서 비교됩니다.
- **배치 처리**: 대량 인덱싱은 배치로 묶어 처리량을 높입니다. GPU가 임베딩에 활용됩니다.

### 2.3.1 멀티모달 인제스트

문서가 깨끗한 텍스트와 정형 표 위주라면, 광학 문자 인식(OCR)으로 텍스트를 뽑은 뒤 텍스트 임베딩하는 방식(OCR-then-text)이 정확도·비용·속도 면에서 대체로 유리합니다. 반면 레이아웃에 의존하는 문서(표, 다이어그램, 차트)나 텍스트 레이어가 없는 스캔본은 멀티모달(네이티브 비전) 임베딩이 적합합니다. OCR은 2차원 레이아웃을 1차원 텍스트로 선형화하면서 정보 손실(표 셀 병합, 캡션 분리 등)이 크기 때문입니다. ([BigData Boutique](https://bigdataboutique.com/blog/multimodal-rag-retrieval-over-images-pdfs-and-text), [Weaviate](https://weaviate.io/blog/multimodal-guide))

하이브리드 라우팅을 권장합니다. 페이지당 추출된 텍스트가 임계값 미만이면 고정확 OCR로, 복잡한 페이지는 비전 임베딩으로 라우팅합니다. OCR 품질이 RAG 성능의 천장을 좌우하므로, 라우팅 판정과 OCR 정확도에 공들일 만합니다.

아키텍처 선택지는 세 가지로 정리됩니다 — 단순성·정확도·비용의 트레이드오프입니다. ([BigData Boutique](https://bigdataboutique.com/blog/multimodal-rag-retrieval-over-images-pdfs-and-text))

- 캡션 생성 후 인덱싱(caption-and-index): 가장 단순.
- 통합 비전 임베딩: 이미지를 그대로 임베딩.
- 페이지 이미지 + 지연 상호작용(page-as-image + late interaction, ColPali 계열): 정확도 우선.

pgvector는 컬럼당 차원이 고정이므로 차원을 사전에 확정해야 합니다. 모델이 마트료시카 표현 학습(MRL, Matryoshka Representation Learning)을 지원하면, 큰 출력 벡터의 앞부분만 잘라 저장해 비용을 줄일 수 있습니다. 모달리티별로 별도 테이블이나 컬럼으로 분리하는 편이 자연스럽고, 단일 공유 벡터 공간이 필요하면 같은 모델로 정렬된 임베딩을 한 컬럼에 모읍니다. ([Weaviate](https://weaviate.io/blog/multimodal-guide))

메타데이터에는 `modality` 태그와 모달리티별 역참조 키를 둡니다(PDF은 `page_number`·`document_id`, 오디오·비디오는 `start_time`·`end_time`·`chunk_index`). 이로써 원본 신호로 역참조하고 출처를 추적할 수 있습니다. 신선도와 접근 규칙(ACL) 정책(2.5)은 모달리티와 무관하게 일관되게 적용합니다. ([Weaviate](https://weaviate.io/blog/multimodal-guide))

## 2.4 pgvector 적재

벡터·원문·메타데이터를 함께 저장합니다. pgvector 스키마·인덱스 운영의 기준 문서는 ②입니다. 여기서는 RAG 관점의 결정만 짚습니다.

```sql
-- 개념 예시 (실제 스키마/차원/연산자는 ② 참조)
CREATE TABLE kb_chunks (
  id          bigserial PRIMARY KEY,
  doc_id      text,
  content     text,                    -- 원문 청크 (답변 근거로 그대로 사용)
  embedding   vector(<dim>),           -- 임베딩 모델 차원과 일치
  metadata    jsonb                    -- 출처·부서·접근등급·작성일
);
```

**인덱스 선택 (HNSW vs IVFFlat)** — 값·파라미터는 ②와 pgvector 버전 문서로 확인:

| 인덱스 | 특성 | 적합 |
|--------|------|------|
| HNSW | 높은 recall, 빠른 검색, 메모리·빌드 비용↑ | 검색 품질 우선, 갱신 빈번 |
| IVFFlat | 빌드 빠름·메모리↓, recall은 튜닝 의존 | 대용량·빌드 비용 민감 |

> 인덱스 파라미터(HNSW의 `m`/`ef_construction`, IVFFlat의 `lists` 등)와 거리 연산자(코사인/L2/내적)는 임베딩 모델 특성에 맞춰야 합니다. 구체 권장값은 ②의 운영 런북과 pgvector 0.8.x 문서를 기준으로 하세요.

## 2.5 재인덱싱·증분 갱신

- **증분**: 문서 추가/수정 시 해당 doc_id의 청크만 삭제 후 재삽입. 메타데이터에 버전/해시를 두면 변경 감지가 쉽습니다.
- **권한(ACL) 재동기화**: 원본 시스템의 문서 권한(ACL)이 바뀌면 본문이 그대로여도 인덱스의 접근등급 메타데이터(2.1)를 재동기화해야, 검색 단계 권한 필터(03·05)가 최신 권한을 반영해 권한 회수된 문서가 노출되지 않습니다.
- **전면 재인덱싱 트리거**: 임베딩 모델 변경, 청킹 전략 변경, 차원 변경 시. 무중단을 원하면 **새 테이블에 빌드 후 스왑**.

### 2.5.1 변경 감지

변경 감지는 세 가지 방식의 트레이드오프로 정리됩니다 — 타임스탬프 워터마크, 콘텐츠 해시, 로그 기반 변경 데이터 캡처(CDC, Change Data Capture)입니다. 이 중 로그 기반(데이터베이스의 WAL 또는 binlog를 읽는 방식)이 최저 지연, 삭제를 포함한 전(全) 변경 포착, 변경 순서 보존에서 가장 강합니다. ([Estuary](https://estuary.dev/blog/the-complete-introduction-to-change-data-capture-cdc/))

주의할 함정이 있습니다. `updated_at` 같은 워터마크를 폴링하는 방식은 하드 삭제(레코드가 물리적으로 사라지는 삭제)를 놓칩니다 — 삭제된 행은 `updated_at`이 갱신되지 않기 때문입니다. 따라서 소프트 삭제 컬럼(삭제 플래그)이나 별도 삭제 로그가 필수입니다. ([Estuary](https://estuary.dev/blog/the-complete-introduction-to-change-data-capture-cdc/))

### 2.5.2 삭제·ACL 회수 전파

삭제와 접근 규칙(ACL) 회수는 반드시 인덱스까지 전파해야 합니다. 누락하면 폐기된 문서나 권한이 회수된 문서가 계속 검색 결과에 노출됩니다.

ACL은 청크 메타데이터로 인덱싱하고, 필터링은 벡터 스토어 쿼리 단계에서 수행합니다(early binding, 보안 트리밍). 애플리케이션에서 검색 후 사후 필터링하는 방식은 금지합니다 — 권한 없는 콘텐츠가 일단 검색 파이프라인에 올라오기 때문입니다. pgvector 예시는 다음과 같습니다. ([Truto](https://truto.one/blog/how-to-maintain-document-level-rbac-in-enterprise-rag-pipelines/))

```sql
-- 사용자 그룹과 청크 ACL이 교집합을 가질 때만 반환 (쿼리 단계 필터링)
... WHERE acl_groups && :user_groups
```

권한 변경은 신원 공급자(IdP, Identity Provider)와 짧은 주기로 재동기화해 반영하되, 오래된 ACL(stale ACL)이 최신 ACL(fresh ACL)을 덮어쓰는 레이스 컨디션에 주의합니다. ([ragaboutit](https://ragaboutit.com/the-permission-layer-problem-why-your-enterprise-rag-is-a-security-time-bomb/))

### 2.5.3 멱등 적재와 백필

변경된 청크만 콘텐츠 해시 기반으로 선택 재임베딩하면 비용을 줄일 수 있습니다. 증분 갱신은 upsert로 처리하되, 임베딩 모델·차원·청킹 전략이 바뀔 때는 전체 재인덱싱이 필요합니다(2.5 본문의 전면 재인덱싱 트리거와 일관). ([Medium 재인덱싱](https://medium.com/@kandaanusha/vector-database-reindexing-pipeline-87efa1d1cd19))

멱등성과 중복 제거(디듑)는 결정론적 키(소스 식별자 + 콘텐츠 해시)와 체크포인트로 확보합니다. 변경 데이터 캡처(CDC)는 통상 at-least-once(최소 한 번) 전달이라 재시도 시 중복이 발생하는데, 이를 upsert로 무해화합니다. pgvector 예시는 `(source_id, chunk_index)`에 유니크 제약을 두고 `ON CONFLICT ... DO UPDATE`로 처리하는 형태입니다. ([Estuary](https://estuary.dev/blog/the-complete-introduction-to-change-data-capture-cdc/))

대규모 초기 백필은 다음 순서로 진행합니다: 스냅샷 적재 → 워터마크 또는 로그 위치 기록 → 증분 스트림으로 전환. 명시적 포지션을 기록해 두면 실패 시 그 지점부터 재개할 수 있고, 변경 누락(드리프트)을 막을 수 있습니다. ([AWS Big Data Blog](https://aws.amazon.com/blogs/big-data/build-a-rag-data-ingestion-pipeline-for-large-scale-ml-workloads/))
